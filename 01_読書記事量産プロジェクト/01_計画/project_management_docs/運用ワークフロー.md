---

## 1. 概要

### 1.1 目的

note読書アカウント「あなたの問いに答える一冊を処方する、本の相談室」において、月間16-24本（週4-6本）の高品質な書籍紹介記事を安定して公開する運用体制を確立する。

### 1.2 運用モデル

**バッチ処理モデル**を採用：
- **週末（土日）**: AI（Claude Code + book-article-generator Skill）で3-5本の記事下書きを一括生成
- **平日（月-金）**: 人間が1本/日のペースで編集・公開

### 1.3 成功基準

- ✅ 週4本以上の記事公開（月間16本以上）
- ✅ 記事品質チェックリストで全項目クリア（構造7/7、内容5/5、文体4/4）
- ✅ 1記事あたりの生成時間45分以内
- ✅ 人間の編集時間15分以内/記事

## 3. AI作業：記事下書き生成

### 3.1 事前準備

#### 書籍選定（前週金曜までに実施）

1. `Books/` フォルダに書籍データがあることを確認
2. `01_計画/書籍リスト管理.md` でステータスを「準備中」に更新
3. 次週生成する5冊をリストアップ

**書籍選定基準**：
- ✅ ビジネス書（営業、マネジメント、交渉、システム開発など）
- ✅ note読者層（20-40代ビジネスパーソン）に関連性が高い
- ✅ 具体的な「問い」を設定しやすいテーマ
- ✅ 書籍データが10,000文字以上ある（分析に十分な情報量）

#### 記事テーマ設定

各書籍について、以下を事前に決定：
- **ターゲット読者**: 具体的なペルソナ（例：小型商談から大型商談に転職した営業パーソン）
- **中心的な問い**: 読者の悩み（例：なぜ小型商談のやり方が大型商談では通用しないのか?）
- **3つの問い**: Why→How→Future の階層構造

### 3.2 生成プロセス（Claude Code）

#### 土曜（3本生成）

```bash
# 1. プロジェクトディレクトリに移動
cd "C:\Users\user\Desktop\kotarosan\03_Internal\02_R&D\01_読書記事量産プロジェクト"

# 2. Claude Codeで以下を実行（各書籍45分）
```

**書籍1本あたりの生成手順**：

1. **フェーズ1: 書籍分析**（15分）
   - `Books/{書籍名}.txt` を読み込み（offset/limit活用）
   - ターゲット読者の悩みに関連する箇所を抽出
   - 著者の信頼性、独自性、実用性を確認

2. **フェーズ2: 記事構成設計**（15分）
   - 3つの問い（Why→How→Future）を設計
   - 各問いで使用する具体的エピソードを選定
   - 処方箋の根拠（著者の信頼性、独自性、実用性）を整理

3. **フェーズ3: 記事執筆**（15分）
   - 7セクション構造で記事を生成
   - `02_テスト/sample_articles/{書籍名}_サンプル記事.md` に保存
   - 文字数1500-2000文字を目標（具体例重視の場合2200文字まで許容）

#### 日曜（2本生成 + 品質チェック）

- 午前：残り2本の記事生成（同様の手順）
- 午後：土曜生成の3本 + 日曜生成の2本の品質チェック

### 3.3 品質チェック（AI実施）

生成後、`02_テスト/test_cases/記事品質チェックリスト.md` に従って自己チェック：

**構造チェック（7項目）**:
- [ ] タイトル: 「【カテゴリ】問い?」形式、25-40文字
- [ ] 導入部: 「こんな問いを持っていませんか?」＋3つの箇条書き
- [ ] 書誌情報: 書名と著者が正しく明記
- [ ] 3つの問い: Why→How→Future の階層構造
- [ ] 処方箋の根拠: 著者の信頼性、独自性、実用性
- [ ] 読了後の未来: 具体的な成果・変化を提示
- [ ] まとめ: 3つの実践アクションが明確

**内容チェック（5項目）**:
- [ ] ネタバレ回避: 本の核心的内容は「期待感」を残す程度
- [ ] 具体性: 抽象論ではなく、書籍の具体的エピソード活用
- [ ] 信頼性: 著者の実績・研究データ・採用企業などで裏付け
- [ ] 文字数: 1500-2000文字（具体例重視の場合2200文字まで許容）
- [ ] 問い駆動: 読者の悩み起点で構成

**不合格の場合**: その場で修正し、再チェック

## 4. 人間作業：編集・公開

### 4.1 品質チェック（人間実施 - 日曜午後）

AIの自己チェック結果を再確認：

1. **文体チェック（4項目）**:
   - [ ] 敬体統一: です・ます調で統一されているか
   - [ ] 平易な表現: 専門用語に説明があるか
   - [ ] 対話的: 「あなた」と呼びかけているか
   - [ ] 親しみやすさ: note読者に適した語り口か

2. **Quick Reference（10項目）**: 全項目確認

3. **最終判定**:
   - ✅ **公開OK**: そのまま次の編集工程へ
   - ⚠️ **要修正**: 具体的な修正指示をAIに送信

### 4.2 編集作業（平日 - 15分/本）

#### 月曜〜金曜の各日

1. **軽微な調整**（10分）:
   - 文体の微調整（語尾、接続詞など）
   - 固有名詞の表記統一
   - 誤字脱字チェック

2. **noteへのアップロード準備**（5分）:
   - タイトル最終調整
   - アイキャッチ画像選定（Canva等で作成、または書籍カバー画像）
   - タグ設定（例：#読書 #ビジネス書 #営業術 #note読書）

### 4.3 公開作業（同日）

1. **noteに投稿**:
   - 記事本文をコピー＆ペースト
   - プレビュー確認
   - 公開設定（無料公開 or 有料マガジン）

2. **公開後の管理**:
   - `01_計画/書籍リスト管理.md` のステータスを「公開済」に更新
   - 公開URLを記録
   - 公開日時を記録

## 5. バリエーション展開戦略

### 5.1 1冊から複数記事を生成

テスト結果から、1冊の書籍から2-3本の記事を生成可能なことが実証されました。

**展開パターン**：

| パターン | 説明 | 例（無敗営業） |
|:---|:---|:---|
| **ターゲット読者変更** | 同じ本を異なるペルソナ向けに | 営業向け → マネジメント向け |
| **問いの変更** | 読者の悩みを別角度から | 「なぜ選ばれないのか?」→「なぜ部下の成績にバラつきが出るのか?」|
| **フォーカス変更** | 書籍の別の章・テーマに焦点 | 個人スキル編 → 組織的仕組み編 |

### 5.2 活用タイミング

- **書籍が豊富な分野**: 無理に1冊から複数記事を作らず、別の書籍で記事作成
- **書籍が少ない分野**: バリエーション展開で記事数を確保
- **人気書籍**: 複数角度から紹介することで読者層を拡大

## 6. 品質管理プロセス

### 6.1 定期レビュー

**月次レビュー**（毎月末）:
- 公開記事数の確認（目標：16本以上）
- 読者反応の分析（PV、スキ、コメント数）
- 品質チェックリストの不合格項目分析
- 次月の改善点抽出

**四半期レビュー**:
- 記事生成Skillの改善提案
- 新しい書籍ジャンルの検討
- 運用ワークフローの見直し

### 6.2 品質基準の更新

読者フィードバックに基づき、品質チェックリストを更新：
- 新しいチェック項目の追加
- 合格基準の調整
- 具体例の充実度基準の明確化

## 7. リスク管理

### 7.1 想定リスクと対策

| リスク | 影響度 | 対策 |
|:---|:---|:---|
| **書籍データ不足** | 高 | 事前に5冊分のストックを確保、Books/フォルダを常に監視 |
| **AI生成の品質低下** | 中 | 品質チェックで不合格の場合は即修正、重大な場合は生成やり直し |
| **人間の編集時間不足** | 中 | バッファとして金曜5本目を「余裕がある週のみ」に設定 |
| **note読者の反応低迷** | 低 | 月次レビューで分析、ターゲット読者設定を見直し |

### 7.2 トラブルシューティング

#### 問題：AI生成が品質チェックに不合格

**原因パターン**：
1. 書籍データの情報量不足（10,000文字未満）
2. ターゲット読者設定が曖昧
3. 具体的エピソードが抽出できない

**対処法**：
1. 書籍データの別の箇所を読み込み（offset/limit調整）
2. ターゲット読者を再定義
3. 別の書籍に切り替え

#### 問題：週4本の公開ペースを維持できない

**対処法**：
1. **短期対策**: 過去のバリエーション展開で記事追加（1冊から2本目を生成）
2. **中期対策**: 土日の生成本数を6本に増やす（余裕のある週末のみ）
3. **長期対策**: 書籍データの事前準備を強化
