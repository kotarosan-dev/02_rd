# etika社内向け｜非エンジニアでも担保できるAI生成コード品質ガイド

**前提**: コードを書く作業そのものは、Cursor / Claude Code などのAIでかなり実行できる。  
**問題**: しかし、AIが書いたコードを「本番に出してよい」と判断するには、コードの細部だけでなく、前提・リスク・検証・運用を人間が押さえる必要がある。  
**本書の目的**: コード知識が十分でないメンバーでも、AIに実装を任せる前後で何を確認すればよいかを、etikaの実務手順として定義する。

---

## 1. 基本方針

AI生成コードの品質担保で、非エンジニアが担うべき役割は「コードを読んでバグを見つけること」ではない。

担うべきなのは次の4つ。

1. **何を作るべきかを曖昧にしない**
2. **事故が起きる場所を先に洗い出す**
3. **AIに検証証拠を出させる**
4. **本番投入してよい条件を人間が決める**

コードはAIが書ける。  
しかし、何を守るべきか、何が起きたら困るか、誰に迷惑がかかるか、どこまで検証すれば出してよいかは、AIに丸投げしてはいけない。

---

## 2. AIに任せてよい部分・人間が担保する部分

| 区分 | AIに任せてよい | 人間が担保する |
|---|---|---|
| 仕様 | 文章化の補助、抜け漏れ質問の生成 | 目的、対象ユーザー、業務上の正解 |
| 実装 | コード生成、修正案、テスト案 | 何を変更してよいか、どこは触ってはいけないか |
| セキュリティ | チェックリスト生成、静的解析の補助 | 個人情報・権限・外部公開のリスク判断 |
| テスト | テストコード生成、テスト観点の提案 | 実際に業務上OKか、顧客影響がないか |
| リリース | 手順案、ロールバック案の作成 | 本番投入の可否、投入タイミング、責任者判断 |

etikaの運用では、AIを「速い実装担当」として扱う。  
AIを「責任者」や「最終判断者」として扱わない。

---

## 3. 最初に判定すること

AIにコードを書かせる前に、その作業がどの危険度かを判定する。

### レベルA: 低リスク

例:

- 表示文言の修正
- 社内だけで使う画面の軽微なUI調整
- 既存処理に影響しない小さな補助機能
- 失敗してもすぐ戻せる試作品

必要な確認:

- 変更前後の画面確認
- 既存機能が壊れていないこと
- 変更ファイルの差分確認

### レベルB: 中リスク

例:

- 顧客データを表示・検索・更新する機能
- CRM / Books / Sign / WorkDrive などZoho連携
- 外部APIとの連携
- メール送信、通知、帳票生成
- 業務フローに影響する自動化

必要な確認:

- 仕様メモ
- リスクチェック
- テスト証跡
- ロールバック方法
- 関係者レビュー

### レベルC: 高リスク

例:

- 個人情報・契約情報・請求情報を扱う
- 権限・認証・ログインに関わる
- 本番データを一括更新・削除する
- 顧客へ自動送信する
- 課金、決済、金額計算に関わる
- 外部公開されるWebフォームやAPI

必要な確認:

- レベルBの確認すべて
- セキュリティ観点レビュー
- 本番相当データでの検証
- 実行前バックアップ
- 失敗時の連絡・停止手順
- 原則として複数人承認

## 4. AIに依頼する前の10問

非エンジニアでも、次の10問には答えられる状態にしてからAIに実装させる。

1. **これは誰のための機能か？**
2. **成功した状態は何か？**
3. **絶対に起きてはいけない事故は何か？**
4. **扱うデータに個人情報・金額・契約情報は含まれるか？**
5. **誰が見られて、誰が見られないべきか？**
6. **AIが触ってよいファイル・触ってはいけないファイルはどこか？**
7. **失敗したら元に戻せるか？**
8. **本番投入前に何を見ればOKと言えるか？**
9. **本番投入後に異常をどう検知するか？**
10. **責任者は誰か？**

この10問に答えられない場合、先にコードを書かせない。  
AIに「不足している仕様を質問して」と依頼し、要件を埋めてから実装に入る。

## 5. AIに必ず渡すミニ仕様

AIへの依頼は、最低限この形式にする。

```markdown
# タスク
何を作るか:

# 背景
なぜ必要か:
誰が使うか:

# 成功条件
- 
- 

# 絶対に避けたいこと
- 個人情報を漏らさない
- 既存データを勝手に削除しない
- 既存の業務フローを壊さない

# 扱うデータ
- 個人情報:
- 金額:
- 契約情報:
- 外部送信:

# 変更してよい範囲
- 

# 変更してはいけない範囲
- 

# 検証してほしいこと
- 正常系:
- 異常系:
- 権限:
- データ件数が多い場合:
- 失敗時の戻し方:
```

ポイントは、AIに「作って」とだけ言わないこと。  
AIには、守るべき制約と確認すべき観点まで渡す。

## 6. AI実装中に必ず出させるもの

AIに実装させるときは、コードだけでなく次の成果物も同時に出させる。

### 1. 変更概要

- どのファイルを変えたか
- 何のために変えたか
- 既存機能への影響は何か

### 2. リスク一覧

- データ漏洩リスク
- 誤更新・誤削除リスク
- 権限不備リスク
- コスト増加リスク
- 外部API失敗リスク
- 運用で検知できないリスク

### 3. テスト結果

- 実行したテスト
- 成功した結果
- まだ未確認のこと
- 手動確認が必要なこと

### 4. 本番投入条件

- 何が確認できたら出してよいか
- 何が残っていたら出してはいけないか
- 失敗したらどう戻すか

AIがこれらを出せない場合、実装が動いていても完了扱いにしない。

## 7. 非エンジニア向け 7つの品質観点

### 1. セキュリティ

見るポイント:

- ログインしていない人が見られないか
- 本人以外のデータが見られないか
- 管理者だけができる操作を一般ユーザーができないか
- APIキー、パスワード、トークンがコードやログに出ていないか

AIへの確認例:

> この実装で、他人のデータを見られる可能性がある箇所を洗い出してください。  
> 認証・認可・入力検証の観点で、未対応のリスクを表にしてください。

### 2. データ

見るポイント:

- どのデータを読むか
- どのデータを書き換えるか
- 削除されるデータはあるか
- 失敗した途中状態が残らないか
- 重複作成されないか

AIへの確認例:

> この処理が作成・更新・削除するデータを一覧化してください。  
> 途中で失敗した場合にデータ不整合が起きる可能性を説明してください。

### 3. お金・契約

見るポイント:

- 金額計算に関わるか
- 請求・支払・見積・契約書に関わるか
- 税率、端数処理、通貨、割引の扱いは正しいか
- 誤送信されたときの影響は大きいか

AIへの確認例:

> 金額・契約・請求に関わる処理をすべて列挙し、誤った場合の業務影響を説明してください。

### 4. 権限

見るポイント:

- 誰が実行できるか
- 誰のデータを対象にするか
- 管理者・担当者・閲覧者の違いが守られているか
- Zoho側の権限とアプリ側の権限が食い違わないか

AIへの確認例:

> ユーザー権限ごとに、この機能でできること・できないことを表にしてください。

### 5. 外部連携

見るポイント:

- どの外部サービスに送信するか
- 送信する情報は最小限か
- APIが失敗したらどうなるか
- 同じ処理が二重実行されないか
- レート制限や課金上限はあるか

AIへの確認例:

> 外部APIが失敗・遅延・重複応答した場合の挙動を整理してください。  
> リトライしてよい処理と、リトライしてはいけない処理を分けてください。

### 6. 運用

見るポイント:

- エラーが起きたときに気づけるか
- ログに必要な情報が残るか
- ログに個人情報や秘密情報が出ないか
- 誰が復旧するか
- 復旧手順があるか

AIへの確認例:

> 本番でエラーが起きたとき、誰が何を見れば原因を特定できるかを説明してください。  
> ログに出してよい情報・出してはいけない情報を分けてください。

### 7. AI特有のリスク

見るポイント:

- 存在しないAPIや古い仕様を使っていないか
- もっともらしいが根拠のない説明をしていないか
- テストしていないのに「問題ありません」と言っていないか
- 既存コードの流儀を無視していないか
- 仕様にない機能を勝手に追加していないか

AIへの確認例:

> この実装で、公式ドキュメントや既存コードで確認できていない前提を列挙してください。  
> 推測で書いた可能性がある箇所を明示してください。

## 8. レビュー時の合格条件

非エンジニアのレビューでは、コードの1行1行を理解しようとしなくてよい。  
代わりに、次の証拠が揃っているかを見る。

### 必須証拠

- 仕様メモがある
- 変更対象が説明されている
- 触ってはいけない範囲が守られている
- リスク一覧がある
- テスト結果がある
- 未確認事項が明示されている
- ロールバック方法がある
- 本番投入条件が明確

### 差し戻し条件

次のどれかに当てはまる場合は、本番投入しない。

- 「たぶん大丈夫」「おそらく問題ない」だけで証拠がない
- テストを実行していない
- 個人情報・金額・契約情報の扱いが曖昧
- 権限チェックの説明がない
- 外部API失敗時の挙動が不明
- ロールバックできない
- AIが変更理由を説明できない
- 仕様にない変更が混ざっている

## 9. 本番投入前チェックリスト

レベルB以上の作業では、本番投入前にこのチェックを行う。

- [ ] 目的と成功条件が明文化されている
- [ ] 変更範囲が限定されている
- [ ] 個人情報・金額・契約情報の扱いを確認した
- [ ] 権限ごとの動作を確認した
- [ ] 正常系を確認した
- [ ] 異常系を確認した
- [ ] 外部API失敗時の挙動を確認した
- [ ] 二重実行・重複作成のリスクを確認した
- [ ] ログに秘密情報が出ないことを確認した
- [ ] テスト結果または手動確認結果が残っている
- [ ] 戻し方が決まっている
- [ ] 本番投入後に何を監視するか決まっている
- [ ] 責任者が承認した

## 10. AIへの標準プロンプト

### 実装前レビュー用

```markdown
これからAIに実装させる予定です。
私はコードに詳しくないため、実装前に危険な点を洗い出してください。

# 作りたいもの

# 使う人

# 扱うデータ

# 絶対に避けたい事故

以下を出してください。
1. 不足している仕様質問
2. セキュリティ・データ・権限・外部連携・運用のリスク
3. 実装前に決めるべきこと
4. 本番投入前に確認すべきこと
```

### 実装後レビュー用

```markdown
以下の変更を本番に出してよいか判断したいです。
コードの細部ではなく、非エンジニアでも確認できる証拠ベースでレビューしてください。

確認してほしいこと:
1. 仕様に合っているか
2. 顧客データ・個人情報・金額・権限のリスク
3. 失敗時に戻せるか
4. テスト証拠は十分か
5. 本番投入してよい条件
6. 出してはいけない未解決事項
```

### AIに嘘をつかせない確認用

```markdown
あなたの回答のうち、実ファイル・実行ログ・公式ドキュメントで確認できたことと、推測を分けてください。
推測は「推測」と明記してください。
未確認の項目は「未確認」と書き、確認方法を提案してください。
```

## 11. etikaでの運用ステップ

### Step 1: 依頼をミニ仕様にする

担当者は、AIに依頼する前に「目的・成功条件・避けたい事故・扱うデータ」を書く。  
この時点では技術用語は不要。業務上の正解を言語化する。

### Step 2: 危険度をA/B/Cで判定する

個人情報、金額、契約、権限、外部送信、本番データ一括処理があれば、原則レベルB以上。  
複数に該当する場合はレベルCとして扱う。

### Step 3: AIに実装計画を出させる

いきなりコードを書かせず、まず変更範囲・リスク・テスト方針を出させる。  
計画が曖昧なら差し戻す。

### Step 4: 実装させる

実装時は、変更範囲を限定する。  
「ついでに直す」「全体をきれいにする」は原則禁止。別タスクに分ける。

### Step 5: AIに自己レビューさせる

実装後、AIに次を出させる。

- 変更ファイル一覧
- 仕様との対応
- リスクと対策
- テスト結果
- 未確認事項
- ロールバック方法

### Step 6: 人間が証拠を見る

人間はコードの細部ではなく、証拠を確認する。  
証拠がないものは「未確認」として扱う。

### Step 7: 本番投入または差し戻し

本番投入してよいのは、次の状態になったときだけ。

- 実装内容が仕様に合っている
- 重大リスクが説明されている
- テストまたは手動確認の証拠がある
- 戻し方がある
- 責任者が承認している

## 12. 最初の30日で作るもの

etika社内でまず整えるべきものは、大きな仕組みではなく、毎回使える小さな型。

- [ ] AI実装依頼テンプレート
- [ ] リスク判定A/B/C表
- [ ] 本番投入前チェックリスト
- [ ] 実装後レビュー用プロンプト
- [ ] ロールバック手順テンプレート
- [ ] 個人情報・金額・権限が絡む作業の承認ルール

これらがあれば、コード知識が薄いメンバーでも「何を確認すればよいか」が揃う。

- Zoho連携、外部API、個人情報、金額処理ごとの個別チェックリストを作る
- AI生成コードの本番投入判断を記録として残す

## 14. まとめ

AI時代に必要なのは、全員がコードを深く読めるようになることではない。  
必要なのは、AIが作ったものをそのまま信じず、次の問いを毎回立てられること。

- 何を作るべきだったのか
- 何が起きたら困るのか
- どのデータを触るのか
- 誰の権限で動くのか
- 失敗したら戻せるのか
- テストした証拠はあるのか
- 本番に出してよい条件は何か

etikaの品質保証は、AIにコードを書かせないことではない。  
AIに速く書かせたうえで、人間が前提・リスク・証拠・責任を握ることにある。
