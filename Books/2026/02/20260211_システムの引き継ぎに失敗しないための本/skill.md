# システム引き継ぎ管理フレームワーク

## 概要

システム引き継ぎにおける3つの構造的問題（モチベーション問題、情報格差、スキルレベル不一致）を踏まえ、準備・実施・事後の全工程を体系的に管理するフレームワーク。担当者の退職、ベンダー変更、アウトソーシング化など、あらゆる引き継ぎシナリオに適用可能。

## 使用場面（When to Use）

以下のいずれかの状況が発生した場合に適用する。

- **担当者退職・異動**: 社内のIT担当者やシステム管理者が退職・異動する際の後任への引き継ぎ
- **ベンダー変更**: システム開発・運用保守を委託している外部ベンダーを変更する場合
- **アウトソーシング化**: 社内運用から外部委託に切り替える場合、またはその逆
- **一人情シスの退職**: 社内に一人しかいないIT担当者が退職する緊急ケース
- **炎上案件の引き継ぎ**: 品質・スケジュール・予算に問題を抱えたプロジェクトの引き継ぎ
- **未完成品の引き継ぎ**: 開発途中のシステムを別のチーム・ベンダーに引き継ぐ場合
- **予防的整備**: 引き継ぎリスクを軽減するための平時のドキュメント整備・ナレッジ管理

## フレームワーク/プロセス

### 全体構造: 3フェーズ・9ステップ

```
Phase 1: 準備（Plan）
  Step 1: 構造的問題の認識と対策
  Step 2: 引き継ぎ書の作成
  Step 3: システム全体像の可視化

Phase 2: 実施（Do）
  Step 4: 事前準備と体制構築
  Step 5: システム理解の推進
  Step 6: システム解析QA
  Step 7: 並行稼働

Phase 3: 事後（Check & Act）
  Step 8: 完了確認
  Step 9: 安定運用への移行
```

### 3つの構造的問題と対策マトリクス

| 構造的問題 | 内容 | 対策方針 |
|---|---|---|
| モチベーション問題 | 引き継ぎ元に引き継ぎのインセンティブがない | 契約上の引き継ぎ義務の明確化、引き継ぎ完了ボーナスの設定、第三者によるファシリテーション |
| 情報格差 | 引き継ぎ先が「何を聞くべきかわからない」 | システム解析QAの実施、引き継ぎ書フォーマットによるチェックリスト化、段階的な知識移転 |
| スキルレベル不一致 | 引き継ぎ元と先の技術力・方針の差異 | 事前のスキルアセスメント、並行稼働期間の確保、技術キャッチアップ計画の策定 |

## 具体的な実行手順

### Phase 1: 準備

#### Step 1: 構造的問題の認識と対策

1. 引き継ぎの種類を特定する（退職/異動/ベンダー変更/アウトソーシング化）
2. 3つの構造的問題のうち、どれがどの程度深刻かを評価する
3. 引き継ぎ元のモチベーション状況を確認し、協力を得るための施策を決定する
4. 引き継ぎスケジュールの概要を策定する

#### Step 2: 引き継ぎ書の作成

以下の項目を網羅する引き継ぎ書を作成する。

- **システム基本情報**: システム名、目的、利用者数、稼働開始時期
- **技術スタック**: 使用言語、フレームワーク、DB、ミドルウェア
- **アカウント情報**: 管理画面、サーバー、クラウドサービス等のアクセス情報
- **定期作業一覧**: バッチ処理、バックアップ、監視設定
- **障害対応手順**: よくある障害パターンとその対処法
- **連絡先一覧**: 関連部署、外部ベンダー、緊急連絡先

#### Step 3: システム全体像の可視化

- **システム概要図**: システム間の連携関係を鳥瞰的に示す図
- **サーバー構成図**: 物理/仮想サーバー、ネットワーク構成の図
- **外部連携一覧**: API連携、データ連携、外部サービスとの接続点の一覧
- **ドキュメント棚卸し**: 既存ドキュメントの存在確認と不足箇所の特定
- **業務フロー整理**: システムが支える業務プロセスの可視化

### Phase 2: 実施

#### Step 4: 事前準備と体制構築

1. 引き継ぎプロジェクトの体制を確定する（責任者、担当者、スケジュール）
2. 引き継ぎ先が必要なアクセス権限を取得する
3. 開発・テスト環境を引き継ぎ先が利用可能な状態にする
4. コミュニケーション手段とミーティング頻度を取り決める

#### Step 5: システム理解の推進

1. 引き継ぎ書に基づいてシステムの全体像を把握する
2. 実環境でシステムを操作し、動作を確認する
3. ソースコードの構成を理解する
4. 不明点をリスト化し、次ステップのQAに備える

#### Step 6: システム解析QA

暗黙知を形式知に変換するための体系的な質問プロセス。

1. Step 5で洗い出した不明点を質問リストにまとめる
2. 引き継ぎ元に対してQAセッションを実施する
3. 回答を文書化し、引き継ぎドキュメントに反映する
4. 新たに発生した疑問点について追加QAを実施する
5. すべての質問が解消されるまで繰り返す

**QAで重点的に確認すべき領域:**
- ドキュメントに書かれていない運用ルール
- 過去に発生した重大障害とその対処法
- システム固有の癖や注意点
- 今後予定されている改修・更新の内容

#### Step 7: 並行稼働

1. 引き継ぎ先が主担当、引き継ぎ元がバックアップの体制で運用を開始する
2. 日常的な運用作業を引き継ぎ先が実施し、引き継ぎ元がレビューする
3. 障害発生時の対応を引き継ぎ先が主導で行い、対応力を検証する
4. 問題が発生した場合は原因を分析し、追加の知識移転を行う
5. 並行稼働期間はシステムの規模と複雑性に応じて設定する（目安: 1〜3ヶ月）

### Phase 3: 事後

#### Step 8: 完了確認

以下の完了基準を満たしているか確認する。

- [ ] すべてのドキュメントが最新の状態で引き継ぎ先に移管されている
- [ ] アクセス権限の移管が完了している
- [ ] 引き継ぎ先が単独で日常運用を遂行できている
- [ ] 障害対応を引き継ぎ先が単独で実施できている
- [ ] 未解決の質問・課題がすべてクローズされている
- [ ] 関係者（利用部門、外部ベンダー等）への連絡先変更通知が完了している

#### Step 9: 安定運用への移行

1. 引き継ぎ元のサポートを段階的に終了する
2. 引き継ぎ完了報告書を作成する
3. 引き継ぎプロセスの振り返りを実施し、改善点を記録する
4. 次回の引き継ぎに備えて、ドキュメントの継続的な更新体制を構築する

## 特殊ケース対応

### 未完成品の引き継ぎ

- 仕様の確定度合いを正確に把握する（確定済み/暫定/未確定）
- テスト状況を確認し、品質リスクを評価する
- 技術的負債のリストを作成し、対処の優先順位を決める
- 残作業の見積もりを引き継ぎ先の視点で再算出する

### 炎上案件の引き継ぎ

- 「火消し」と「引き継ぎ」の優先順位を明確に判断する
- 現状の問題点を洗い出し、影響度と緊急度でトリアージする
- 顧客との信頼関係再構築を最優先課題として扱う
- 段階的な安定化計画を策定し、短期目標と中期目標を設定する

### 一人情シスの退職

- 退職までの期間で移転可能な知識量を見極める
- 外部ベンダーの活用を含む代替体制を早期に検討する
- 最低限のドキュメント（アカウント情報、障害対応手順、連絡先一覧）を優先的に整備する
- 退職後の問い合わせ可能期間について事前に合意する

## 入力/出力

### 入力（引き継ぎ開始時に必要な情報）

| 項目 | 説明 |
|---|---|
| 引き継ぎの種類 | 退職/異動/ベンダー変更/アウトソーシング化 |
| 対象システムの範囲 | 引き継ぎ対象となるシステム・サービスの一覧 |
| スケジュール制約 | 引き継ぎ完了までの期限 |
| 体制情報 | 引き継ぎ元・先の担当者と役割 |
| 既存ドキュメント | 現時点で存在するシステム関連文書 |

### 出力（引き継ぎ完了時の成果物）

| 成果物 | 説明 |
|---|---|
| 引き継ぎ書（完成版） | システムの全情報を網羅した引き継ぎドキュメント |
| システム概要図 | システム間連携を示すアーキテクチャ図 |
| サーバー構成図 | インフラの物理/論理構成図 |
| 外部連携一覧 | API連携、外部サービス接続の一覧表 |
| アカウント管理台帳 | 移管されたすべてのアカウント情報 |
| 障害対応マニュアル | 障害パターンと対処手順の一覧 |
| QA記録 | システム解析QAの全質問と回答の記録 |
| 引き継ぎ完了報告書 | 引き継ぎの実施内容と完了状況の報告 |

## 注意点

- **引き継ぎ工数を甘く見積もらない**: 引き継ぎには通常の想定よりも多くの工数がかかる。システム規模や複雑性に応じて十分な期間を確保すること
- **モチベーション問題を放置しない**: 引き継ぎ元の協力が得られない場合、引き継ぎの品質は大幅に低下する。契約条件やインセンティブ設計で対処する
- **暗黙知の移転を意識する**: ドキュメントに書かれていない知識こそが引き継ぎの成否を分ける。QAセッションを通じて積極的に引き出す
- **並行稼働期間を省略しない**: コスト削減のために並行稼働を短縮・省略すると、引き継ぎ後の障害リスクが大幅に増加する
- **平時からの準備が最重要**: 引き継ぎが発生してからドキュメントを作成するのでは遅い。日常的にドキュメントを最新に保つ運用体制を構築すること
- **ベンダーロックインへの注意**: 特定のベンダーに依存しすぎると引き継ぎのハードルが上がる。標準技術の採用やドキュメント整備で対策する
- **法務・契約面の確認**: ベンダー変更時は知的財産権、ソースコードの帰属、NDA等の法務事項を事前に確認する
