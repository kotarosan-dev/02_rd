# データスチュワードシップ実践フレームワーク

## 基本原則

1. **データは戦略的資産**: データは目録化し、所有し、マネジメントし、理解しなければならない企業資産である
2. **スチュワードシップはガバナンスの運用面**: データスチュワードシップとはデータガバナンスの日常業務が行われる場であり、人材・組織・プロセスで構成される
3. **説明責任の明確化**: データを所有するビジネス機能ごとに、データの意味・品質・ルールについて意思決定する権限者（スチュワード）を任命する
4. **メタデータ中心主義**: データスチュワードシップの本質はメタデータスチュワードシップであり、定義・ルール・リネージの品質管理が中核活動となる
5. **文化変革の必要性**: スチュワードシップの導入は技術施策ではなく組織文化の変革であり、経営陣の支持と段階的な行動変容が必要である

## Instructions

### フェーズ1: ガバナンス体制の設計

1. **三層ガバナンス組織を構築する**: データガバナンスのための組織体制を設計
   - 実行運営委員会: 上級経営陣によるスポンサーシップと方針承認
   - データガバナンス委員会: ビジネス機能の代表者（データガバナー）による意思決定
   - データスチュワードシップ評議会: ビジネスデータスチュワードによる実務運営

2. **データガバナンスプログラム・オフィス（DGPO）を設置する**: スチュワードシップの調整・支援を担う常設組織
   - データガバナンスマネジャーを任命（プログラム全体の管理責任者）
   - エンタープライズデータスチュワードを配置（全社横断的な調整役）
   - DGPOはIT部門ではなくビジネス部門に所属させる
   - スポンサーとなるビジネス部門の幹部に報告ラインを設定

3. **スチュワードの類型を理解し、適切に任命する**: 4種類のスチュワードを目的に応じて配置
   - ビジネスデータスチュワード: データの意味・品質・ルールの意思決定権者。ビジネス機能ごとに任命
   - テクニカルデータスチュワード: ITシステム・物理データの専門家。アプリケーション/データストアごとに任命
   - プロジェクトデータスチュワード: 個別プロジェクトにおけるデータガバナンスの代表者
   - オペレーショナルデータスチュワード: 日常的なデータ品質の監視・是正を担当

4. **RACIマトリックスを策定する**: 各スチュワード類型の役割と責任を明文化
   - R（Responsible）: 実行責任者
   - A（Accountable）: 説明責任者
   - C（Consulted）: 相談対象者
   - I（Informed）: 情報共有対象者

**完了条件**: 三層ガバナンス組織が設計され、DGPOが設置され、スチュワードの類型と任命基準が定義され、RACIマトリックスが策定されている。

### フェーズ2: 支持の獲得と出発点の把握

1. **経営陣と現場の双方から支持を得る**: スチュワードシップ導入のメッセージを準備し伝達
   - 経営陣向け: データ品質問題によるビジネスインパクト（コスト、リスク、機会損失）を定量化
   - 現場向け: データに関する日常的な苦痛（定義の不一致、品質問題、重複作業）を共感的に示す
   - ROIの具体例を準備（例: 住所データ改善によるコスト削減、パレット最適化による輸送効率化）
   - コミュニケーション計画を策定し、ターゲットを絞ったメッセージを発信

2. **組織文化を診断する**: 現在の意思決定スタイルとデータに対する態度を分析
   - コンセンサス重視の文化では経営陣による公式支持がより重要
   - ビジネス機能とビジネスユニットの組織構造を把握（組織図の確認）
   - ビジネスデータスチュワードはビジネスユニットではなくビジネス機能に紐づけて安定性を確保

3. **既存資産を棚卸しする**: ゼロから始めず、既に存在する知識と仕組みを発見
   - データ: どんなデータがあるか、どこから来るか、インフォメーションチェーンの把握
   - メタデータ: アナリストが個人で収集している定義、導出方法、スプレッドシート等を発見
   - データ品質: 既存のデータ品質ルール、問題記録、プロファイリング結果を収集
   - プロセス: 既に確立されている反復可能なプロセスを発見・文書化
   - ツール: 既存のETLツール、BIツール、コラボレーションツールのライセンス確認

4. **スモールスタートの領域を選定する**: 最初に着手すべきビジネス機能を特定
   - 財務部門: 標準化された定義と厳格なガバナンスに慣れている
   - マーケティング/販売部門: データ品質の苦痛を定量化しやすい
   - 注目度の高いプロジェクトに連携してROIを早期に実証

**完了条件**: 経営陣と現場の支持が得られ、組織文化が診断され、既存資産が棚卸しされ、スモールスタート領域が選定されている。

### フェーズ3: キービジネスデータエレメント（KBE）のスチュワード

1. **KBEを選定する**: ガバナンスすべき重要なデータエレメントを優先順位付け
   - 財務報告に使用されるデータエレメント
   - リスクと規制報告に使用されるデータエレメント
   - 企業幹部が紹介するビジネスデータエレメント
   - 注目度の高いプロジェクトで使用されるデータ
   - ビジネスデータスチュワードが推薦するデータ

2. **KBEのメタデータを収集・整備する**: データをガバナンスされた状態にするための情報を文書化
   - 標準化されたビジネス名: 社内統一の呼称を定義
   - 標準化されたビジネス定義: 曖昧さのない厳密な定義を1つだけ策定
   - 導出ルール: 計算・導出されるデータの算出方法を具体的に記述
   - 物理的な所在: ビジネスデータエレメントと物理データエレメント（テーブル/カラム）のマッピング
   - データ品質ルール: 文脈に合った品質要件（フォーマット、範囲、有効値、パターン）
   - 作成ルール: エンティティ作成時の必須条件
   - 使用ルール: データを使用できる目的と使用できない目的の規定
   - 責任者: そのデータに関する権限者と意思決定者の特定

3. **良いビジネス定義を作成する**: 曖昧さのない、測定可能で検証可能な定義を策定
   - 循環参照を避ける（定義の中で定義対象の用語を使わない）
   - 類似する用語との違いを明確に区別
   - 略語を使用しない
   - 複数の概念を1つの定義に混在させない（分割して別のKBEを定義）

4. **ビジネス用語集を構築・維持する**: KBEとそのメタデータを一元管理
   - 検索可能で、アクセス権が適切に設定された共有リソースとして構築
   - メタデータリポジトリと連携させ、技術メタデータとビジネスメタデータを統合
   - 承認ワークフローを設定し、変更管理プロセスを確立

**完了条件**: KBEが選定・優先順位付けされ、各KBEのメタデータが収集・文書化され、ビジネス用語集が構築されている。

### フェーズ4: データ品質の改善

1. **データ品質ルールを策定する**: KBEごとにテスト可能な品質ルールを定義
   - シンプルカラム内容ルール: NULLチェック、有効値チェック、型チェック
   - カラム間バリデーションルール: フィールド間の整合性チェック
   - テーブル間バリデーションルール: 外部キー整合性、クロスシステム整合性
   - ビジネスステートメント: ビジネスロジックに基づく複合的な検証

2. **データプロファイリングを実施する**: データの実態を把握し品質問題を発見
   - プロファイリングツールを使用してデータの分布、異常値、パターンを分析
   - ルール適合性を測定し、違反の頻度と深刻度を評価
   - ルール適合性が低い場合はルール自体の見直しも検討（メタデータ品質の問題の可能性）

3. **根本原因分析と改善を実行する**: データ品質問題の源泉を特定し是正
   - ビジネスデータスチュワードとテクニカルデータスチュワードの協働による分析
   - ソースシステムでの入力バリデーション強化
   - ETLプロセスでのデータクレンジングルール実装
   - ロード時のデータ品質強制（停止、スキップ、ログ、フラグ付きロードの4種類から選択）

4. **メタデータ品質を評価・改善する**: スチュワードシップの成果物自体の品質管理
   - メタデータ品質の評価軸: 完全性、妥当性、アクセス性、適時性、一貫性、有用性、一意性、正確性、整合性
   - メタデータのプロファイリング: ビジネス用語集やリポジトリのメタデータをデータと同様に検査
   - 定義の品質基準への適合性を定期的にレビュー

**完了条件**: データ品質ルールが策定され、プロファイリングが実施され、根本原因が分析・是正され、メタデータ品質の評価体制が整っている。

### フェーズ5: 日常運用の確立

1. **イシューログを運用する**: データに関する問題を体系的に管理
   - 中央集中型のイシューログ（課題管理表）を設置
   - 問題の記録、分類、優先順位付け、担当スチュワードへの割当て、進捗追跡
   - 定期レビューによる問題のエスカレーションと解決促進

2. **定例会議とワーキンググループを運営する**: スチュワード間の協力体制を確立
   - データスチュワードシップ評議会の定例会議
   - ビジネスデータスチュワードとDGPOの定例会議
   - 特定課題に対するワーキンググループの活用
   - 対話型フォーラムの活用（データアナリストとの知見共有）

3. **リファレンスデータを管理する**: 有効値リストとコード体系の維持管理
   - 既存の有効値を文書化し周知
   - 新しい有効値の必要性評価と承認プロセス
   - システム間のコード値の調整（クロスウォーク）の管理
   - 未承認値の使用検出メカニズムの構築

4. **ワークフローによるプロセス自動化を推進する**: 承認・通知・エスカレーションを自動化
   - メタデータの変更承認ワークフロー
   - データ品質ルール違反時の通知ワークフロー
   - 新しいKBE追加時のレビュー・承認ワークフロー

5. **コミュニケーション計画を実行する**: スチュワードシップの成果と進捗を組織に周知
   - データスチュワードシップポータルまたはWikiの構築
   - ターゲットを絞ったコミュニケーション（経営陣、ビジネスユーザー、IT部門それぞれに最適化）
   - 成果事例の定期的な発信

**完了条件**: イシューログが運用され、定例会議が機能し、リファレンスデータが管理され、ワークフローが自動化され、コミュニケーション計画が実行されている。

### フェーズ6: 進捗測定と成熟度向上

1. **ビジネス成果指標を測定する**: スチュワードシップがビジネスに与える効果を定量化
   - データ品質改善によるコスト削減額
   - データ品質問題に起因するビジネスリスクの低減
   - データ関連プロジェクトの成功率向上
   - 規制遵守の改善

2. **運用指標を測定する**: スチュワードシッププログラムの受入状況とパフォーマンスを追跡
   - ガバナンスされたKBEの数と割合
   - メタデータの完全性・品質スコア
   - イシューの平均解決時間
   - スチュワードの参加率・活動量
   - スコアカードの作成と定期的な報告

3. **成熟度モデルで現状を評価する**: 5段階のレベルと4つの評価軸で組織を診断
   - レベル1（初期）: 非公式で属人的、プロセスなし
   - レベル2（戦術的）: 一部で取り組み開始、限定的なプロセス
   - レベル3（明確な定義）: 正式なプロセスと体制が確立
   - レベル4（戦略的）: 全社的に統合され、ビジネス価値を創出
   - レベル5（最適化）: 継続的改善が文化として定着
   - 評価軸: 組織の認知、役割と体制、標準ポリシー・プロセス、価値創造

4. **ギャップを特定し是正策を策定する**: 現在の成熟度と目標レベルの差分を分析
   - 各評価軸のスコアを比較し、最も改善効果の高い領域を特定
   - 是正策をロードマップに反映し、段階的に実行
   - 定期的に成熟度を再評価し、進捗を追跡

5. **データドメインベースのスチュワードシップへの進化を検討する**: 成熟した組織向けの高度な体制
   - ビジネス機能ベースからデータドメインベースへの移行
   - データドメイン評議会の設置（リードデータスチュワードを中心とした複数スチュワードのチーム）
   - ドメイン横断的なデータ標準化と品質管理

**完了条件**: ビジネス成果指標と運用指標が測定され、成熟度評価が実施され、ギャップ分析に基づく是正策が策定されている。

## チェックリスト

### 導入時チェックリスト
- [ ] 経営陣のスポンサーが確保されている
- [ ] 三層ガバナンス組織が設計されている
- [ ] DGPOが設置され、データガバナンスマネジャーが任命されている
- [ ] ビジネスデータスチュワードの任命基準が定義されている
- [ ] 既存のデータ、メタデータ、ツール、プロセスの棚卸しが完了している
- [ ] スモールスタート領域が選定されている
- [ ] コミュニケーション計画が策定されている

### 運用時チェックリスト
- [ ] KBEが選定され、メタデータが収集・文書化されている
- [ ] ビジネス用語集が構築され、アクセス可能な状態で維持されている
- [ ] データ品質ルールが策定され、定期的にプロファイリングが実施されている
- [ ] イシューログが中央集中型で運用されている
- [ ] 定例会議が機能し、スチュワード間の協力が行われている
- [ ] ワークフローによるプロセス自動化が導入されている
- [ ] 進捗指標が定期的に測定・報告されている

### 成熟度向上チェックリスト
- [ ] 成熟度評価が実施され、現在のレベルが特定されている
- [ ] 目標成熟度レベルが設定されている
- [ ] ギャップ分析に基づく是正策のロードマップが策定されている
- [ ] ビジネス成果指標でROIが実証されている
- [ ] データドメインベースへの進化が検討されている
