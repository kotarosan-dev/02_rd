## 読書ナビゲーション

本書は「DXを進めようとしたとき、組織の中に存在する分断にどう向き合うか」という問いに正面から答える実践書である。DX戦略論でも技術論でもなく、現場と経営が一致して動ける状態を作るための「ジャーニー（旅）」の設計図を示す。

構成は大きく二部に分かれる。前半（第1・2章）がDXを取り巻く日本の現状認識と、組織変革の羅針盤となる「デジタルトランスフォーメーション・ジャーニー」の全体設計。後半（第3〜9章）が4つの段階それぞれの現場実践と変革戦略の詳解となる。

特に第6章（仮説検証型アジャイル開発）と第8章（アジャイルブリゲード）は本書の核心であり、「正しいものを正しく作る」サイクルと「組織の水平的な分断を越境するチーム設計」の具体論が詰まっている。CDOや戦略文書が先にある必要はない。「どこからでも始められる」というのが著者の一貫したメッセージだ。

## コアフレームワーク

### DXの3段階構造（デジタイゼーション→デジタライゼーション→DX）

```
Digitization（デジタイゼーション）
 ↓ アナログ・物理データのデジタルデータ化（紙→電子）
Digitalization（デジタライゼーション）
 ↓ 個別業務・プロセスのデジタル化（自動化・効率化）
Digital Transformation（DX）
 ↓ 組織横断の業務変革 + 顧客起点の価値創出 + ビジネスモデル変革
```

第1・2段階はあくまで組織内カイゼン。第3段階（DX本丸）は「提供価値の変革」と「組織の変革」を同時に起こすものであり、既存業務の延長では到達できない。経産省DX推進ガイドライン（2018年）の定義：「データとデジタル技術を活用して…競争上の優位性を確立すること」。

### デジタルトランスフォーメーション・ジャーニーの4段階

```
第1段階：業務のデジタル化
  └─ コミュニケーションのデジタル化（第3章）
  └─ デジタル化の定着・展開（第4章）
         ↓
第2段階：スキルのトランスフォーメーション
  └─ 探索のケイパビリティ獲得（第5章）
         ↓
第3段階：ビジネスのトランスフォーメーション
  └─ 仮説検証型アジャイル開発（第6章）
  └─ 垂直上の分断を越境（第7章）
         ↓
第4段階：組織のトランスフォーメーション
  └─ 水平上の分断を越境（第8章）
  └─ 組織のジャーニーを続ける（第9章）
```

4段階は直線的に進むのではなく、重なりを持ちながら並行的に進める。最初から組織変革に注力するのではなく、まず業務・事業のDXを通じた実績で地に足のついた変革を積み上げる。

### 二項動態：対立から提携へ

| 概念 | 内容 |
|------|------|
| **深化（1.0）** | 既存事業の効率化・洗練化・カイゼン。PDCA、ウォーターフォール、計画駆動 |
| **探索（2.0）** | 新たな価値創出のための仮説検証・実験。OODA、アジャイル、仮説駆動 |
| **二項対立** | 1.0か2.0かの二択。片方を否定し排除しようとする |
| **二項動態** | 状況に応じて1.0・2.0を使い分ける動的なバランス（野中郁次郎提唱）|

「PDCAはもう古い」式の二項対立思考は誤り。深化のケイパビリティは既存事業でそのまま活きる。探索的局面ではOODAを前面に出しつつ、確実な遂行が必要な部分ではPDCAを適用する。**どちらかを捨てるのではなく、どちらをいつ使うかの「二項動態」が核心。**

### 適応課題と技術的課題の分類

| | 技術的課題 | 適応課題 |
|--|-----------|----------|
| **解決方法** | 知識・ツール・スキルで解決可能 | ものの見方・評価基準の変革が必要 |
| **DXでの例** | クラウド移行技術を学ぶ | アジャイルへの抵抗・PDCAへの依存 |
| **アプローチ** | 技術導入・研修 | アライアンス（提携的発想）・段階設計 |

DXが頓挫する主因は技術課題ではなく適応課題。「なぜ変えるのか？」を問い続ける組織は深化の評価基準に縛られている。問い直すべきは「なぜ変えないのか？」。

### 仮説検証型アジャイル開発の5ジャーニー

```
第1ジャーニー：価値探索（課題検証）
  └─ 仮説キャンバス作成 → 顧客インタビュー（10名1セット）→ 仮説修正
第2ジャーニー：価値探索（ソリューション検証）
  └─ プロトタイプ制作 → プロトタイプ検証 → MVP特定
第3ジャーニー：スプリントゼロとリリースプランニング
  └─ アーキテクチャ設計 → バックログ整備 → リリース計画
第4ジャーニー：MVP開発
  └─ スクラムによるプロダクト開発
第5ジャーニー：MVP検証
  └─ PSfit/PMfit評価 → 継続・方針転換・停止の判断
```

ターンアラウンド（働きかけ→反応→理解修正）を最短化することが肝要。「早く失敗して学びを得る」の実装形。

### アジャイルブリゲード：水平の分断を越境するチーム

```
アジャイルディビジョン（戦略層）
  ├─ 経営人材
  ├─ ブリゲードリード
  └─ DX推進メンバー
         ↓ ミッション付与
アジャイルブリゲード（越境チーム）
  ├─ 専門人材（仮説検証/アジャイル/クラウドAI等）
  ├─ 経営人材（後ろ盾）
  ├─ 内部人材
  └─ 外部人材（招聘）
         ↓ 越境
既存事業部門（越境先）
```

**アジャイルブリゲード**は「出島戦略の限界を超えるための越境チーム」。探索の専門性（仮説検証・アジャイル）と希少技術（クラウド・AI・データ基盤）の2本立てで、組織の水平分断を人力で繋ぐ。**アジャイルディビジョン**は複数ブリゲードを束ね、戦略のフロー効率性とリソース効率性を同時に管理する。

## 章別エッセンス

### 第1章：DX1周目の終わりに

**核心命題**: 日本は国際競争力ランキングで1992年首位から2020年34位まで転落した「衰退途上国」であり、過去の栄光も未来への危機感も実感しにくい世代が組織を担っている。DXという言葉は組織変革の旗印として使うべき「絶好の機会」だ。DXレポート2（2020年）では調査企業の9割以上がDXに未着手か散発的な実施にとどまっている。

**事例サマリー**:
- 経産省DXレポート（2018年）が「2025年の崖」を提唱。既存ITシステム放置による経済損失最大12兆円を警告
- DXレポート2（2020年末）：「DX銘柄」選出の一方、企業の9割以上が未対応
- 伊勢神宮式年遷宮：20年周期でのシステム（社殿）更新と技術継承のモデルとして言及

**行動指針**: DXを「システム刷新」と矮小化しない。「組織文化のレガシー化」こそ本質と認識し、変革の旗印としてDXを組織内の共通言語化に使う。

**反直感**: 「DXはバズワードだから使わない」という姿勢が、変革の機運をみすみす逃す原因になっている。言葉に振り回されないためにこそ、意図的に「利用する」スタンスが必要。

### 第2章：デジタルトランスフォーメーション・ジャーニーを描く

**核心命題**: DXには「適応課題」が立ちふさがる。技術的課題（知識・スキルの不足）と異なり、適応課題はものの見方・評価基準自体の変革を要する。「アップデート（更新）」ではなく「アライアンス（提携）」という発想、そして「二項対立」から「二項動態」へのシフトが変革推進の基本スタンスとなる。

**事例サマリー**:
- オライリー・タッシュマン「両利きの経営」の「知の深化と探索」フレームワーク。既存事業は深化に最適化されており探索能力が育っていない
- 出島戦略（江戸幕府の鎖国に由来）：既存組織と新規取り組みを物理的に分離する手法。DXの入り口として有効だが、本土との連携が不可欠

**行動指針**: 段階の設計において目的地（ゴール）を先に置き、バックキャスティングで各段階を描く。各段階の終わりに「ふりかえり」と「むきなおり」を必ず実施する。1ジャーニー＝1〜3か月程度。

**PDCAとOODA**:
- PDCAは「確実に遂行する」仕事の基本。捨てる必要はない
- OODAは「観察→適応→意思決定→行動」。探索局面では短いPDCAサイクルに近い
- **タイムボックス（1周の期間）** と **バランス（計画vs探索）** の2軸でPDCAを状況に合わせて調整する

### 第3章：コミュニケーションのトランスフォーメーション（業務のデジタル化①）

**核心命題**: DXの最初の一手はコミュニケーションの再設計。「ダウンロード型」（バッチ・オフライン・アナログ）から「ストリーミング型」（リアルタイム・オンライン・デジタル）へ移行することが、高頻度トライアルを可能にする探索的活動の土台を作る。

| | ダウンロード型 | ストリーミング型 |
|--|-------------|---------------|
| コミュニケーション | メール・対面会議 | チャット・Web会議 |
| 情報共有 | 紙・物理ファイル | クラウドオンラインファイル |
| プロトコル | 重い（礼儀・段取り必要）| 軽い（短文・非同期OK）|
| 主体性 | PUSH型（送り手主導）| PULL型（受け手主導）|

**3つの変革効果**:
1. **ミーティングのアンバンドル化**：対面会議をチャットで細分化・非同期化。並行進行が可能に
2. **PUSH型からPULL型へ**：コミュニケーションの主導権が受け手に移る。mentionで必要な人のみに届ける
3. **会話のオンライン化**：ブレストや雑談もチャットで代替。偶発的な知識共有が生まれる

**デジタルコミュニケーションの限界**:
- テキストは伝わらないことが多い。3行以内・伝えたいことを最初に・相手がどう受け取るかを想像する
- 「常時接続＝常に返信できる」ではない。相手の状況が見えないため、自分の状態を意識的に発信する必要がある

**変革戦略（組織ポリシーへの3作戦）**: ①正攻法で経営人材を巻き込む ②外部パートナーの環境を使う ③出島（別組織）で別ポリシーを適用する

### 第4章：デジタル化の定着と展開（業務のデジタル化②）

**核心命題**: ツール導入では変化は定着しない。「2つの谷」（導入の谷・定着展開の谷）を越えるためには、チームで仕事をするための「協働の型」を同時に導入する必要がある。協働の型とは「見える化→場作り→一緒にやる」の3段階。

**理解の質のサイクル（ダニエル・キム「成功循環モデル」を改良）**:
```
関係の質 → 理解の質（目標・状況・お互いの理解）
    ↑              ↓
 結果の質 ←  行動の質 ← 思考の質
```

**タスクマネジメントの3つのコツ**:
1. **目標と制約条件の明確化**：ゴールデンサークル（WHY→HOW→WHAT）を描く
2. **抜け漏れなくタスクを挙げる**：チームの複数人の経験を活用し、プロジェクトマネジメント知識体系を観点として使う
3. **ムービングターゲットと認識**：目的自体が変わることを前提に、1〜2週間ごとにゴールデンサークルを見直す

**展開戦略の5ステップ**:
1. 展開先を決める
2. マネジメント層・現場両方にヒアリング（「導入施策の課題」ではなく「展開先の課題」に焦点）
3. 免疫マップで課題整理（改善目標・阻害行動・裏の目標・固定観念の4列）
4. 段階の設計を適用
5. 変革のトレース（関心を保持し続ける）

**事例**: ある組織では営業が社内データ確認のために毎回電話をかけていた。VPN接続は可能なのに「電話の方が早い」という思い込み。ツール導入前にこの行動背景を理解し、日常ツール（LINEビジネス版）から段階的に移行することで抵抗を減らした。

### 第5章：探索のケイパビリティの獲得（スキルのトランスフォーメーション）

**核心命題**: 組織として「探索のケイパビリティ」を獲得するためには、まず個々人と組織のケイパビリティの現状を可視化（棚卸し）してから施策を打つ。探索能力＝仮説検証力（構想）＋アジャイル開発力（実現）の両輪が必要。いきなり全員研修は意味がない。

**日本CTO協会「DXクライテリア」の構成**:
- 5テーマ×8カテゴリ×8項目＝320チェック項目
- チーム・システム・データ駆動・デザイン・経営の5テーマ
- 「超高速な事業仮説検証能力を得ること」を目的に置く

**星取表（スキルマップ）の運用3ポイント**:
1. 月1回「星取作戦会議」でスキル表を更新
2. 個人とチーム両面で星取方針を決める
3. 研修・社内勉強会・パイロットPJ・メンタリングの選択肢を用意

**人材育成の2原則**:
- **イネーブルメントアプローチ**：成果→行動→知識の逆算。体系的網羅研修ではなく、達成すべき成果から必要な学びを絞る
- **行為から学ぶ**：まず動いてみて、その結果のふりかえりから価値観・原則へさかのぼる。2001年以来積み上がったアジャイルの実践知がこの方法で広がってきた

**組織レベルのアセスメント**:
- **DX推進指標**（経産省）：経営とITシステムの2軸、30以上の設問。経営者自らが回答し、「気づき」を得るための対話を促す
- **DX認定制度**（経産省・IPA運営）：「DX Readyな状態」の認定。デジタルガバナンスコードへの準拠が判断基準

### 第6章：仮説検証とアジャイル開発（ビジネスのトランスフォーメーション①）

**核心命題**: DXの核心は「わからないことをわかるようにする」仮説検証型アジャイル開発にある。闇雲な「とにかくやってみる」と仮説検証の違いは「問い」の有無。「ターンアラウンド（働きかけ→反応→理解修正の一周）」を最短化することが、不確実性への対処法。

**PSfit / PMfit の軸**:
- **PSfit**（Problem-Solution-fit）：特定の課題と解決策の一致度。「解決に値する課題があるか」
- **PMfit**（Product-Market-fit）：プロダクトが市場に受け入れられる度合い。「ビジネスになるか」

**仮説キャンバスの主要観点**:
目的・ビジョン・提案価値・顕在課題・潜在課題・代替手段・状況・傾向・評価指標・実現手段・優位性・チャネル・収益モデル・市場規模

**セットベース思考**: 最初から1つのアイデアに決め打ち（ポイントベース）せず、選択肢を広く取り（セット）、検証によって絞り込む。選択の幅＝最大から最小への圧縮が仮説検証型アジャイル開発の全体構造。

**アジャイル開発の9つの意義**（スクラムの意義）:
フィードバック適応、認識共有、問題の早期発見、チーム学習、早期開発開始、結合リスク早期解消、利用開始期間短縮、開発リズム整備、協働育成

**不確実性の飼い慣らし2モード**:
- モード1：仮説検証で不確実性を低減（わかることを増やす）
- モード2：チームの多様性やあえての視点展開で不確実性を高める（わからないことを増やす）→事業の可能性を広げるために必要

### 第7章：垂直上の分断を越境する（ビジネスのトランスフォーメーション②）

**核心命題**: MVP検証が進むにつれ、経営・マネジメント層と現場の間で「関心と情報量の2つのズレ」が生じる（垂直上の分断）。この分断の解消には「CoE（Center of Excellence）」の設置が有効。CoE自体もアジャイル（スクラム）で運営することで、戦略と現場の往復を高速化する。

**MVP検証以降のロードマップ**:
```
第6ジャーニー：UXの推敲（バケツの穴をふさぐ）
  └─ ユーザー行動フローで離脱箇所を特定・改善
第7ジャーニー：PMfitへのチャレンジ（マジョリティへ）
  └─ アーリーアダプター層のシグナルをマジョリティに向けた仮説に転換
第8ジャーニー以降：デュアルトラックアジャイル
  └─ 仮説検証とプロダクト開発を並走させる
```

**CoEの役割**:
- 戦略（経営）と現場（開発チーム）の橋渡し
- プロダクトマネジメントオフィス（PdMO）機能を担う
- アジャイルなCoE：コE自体をスクラムで運営し、ふりかえり・むきなおりを定期化
- CoEの設置遅延が最大のリスク。垂直分断は「絵に描いた餅DX」に直結する

**AARRR指標（事業評価の5段階）**:
Acquisition（獲得）→ Activation（活性化）→ Retention（継続）→ Referral（紹介）→ Revenue（収益）

**CPfit**（Context-Problem-fit）: PSfit前段階。「どういう状況の人が課題を持つのか」コンテキスト自体を検証する概念。

---

### 第8章：水平上の分断を越境する（組織のトランスフォーメーション①）

**核心命題**: 出島戦略の限界は、出島の成果が本土（既存組織）に伝播しないこと。水平上の分断（部署・事業部門間の隔たり）を越えるために「アジャイルブリゲード」を組成し、組織の内外を横断するチームとして機能させる。複数ブリゲードを束ねる「アジャイルディビジョン」が戦略と現場の整合を担う。

**アジャイルブリゲードの4要件**:
1. 探索の専門性（仮説検証・アジャイル）を持つ
2. 専門人材と経営人材、内部人材と外部人材で構成
3. 新規・既存を問わず組織内事業部に越境し協働
4. アジャイルに運営する（ブリゲードバックログ＋ジャーニーマップ）

**立ち上げの5ケース**: ①経営直下 ②情報システム部門 ③DX専任部署 ④事業部門が自ら ⑤現場担当者起点

**ブリッツトラスト**: 最初の信頼関係構築のために一時的に持続可能性を捨て、短期集中で圧倒的な結果を出す作戦。「ブレーキを壊してアクセルだけ踏む」期間を意図的に設定する。

**アジャイルディビジョンの2つの効率性**:
- **フロー効率性**：どれだけ施策が途切れなく実現へと流れるか（戦略と現場の往復ターンアラウンドを短くする）
- **リソース効率性**：希少な専門性が適切に配置されているか

**インセプションデッキ（10項目）**: われわれはなぜここにいるのか・エレベーターピッチ・パッケージデザイン・ご近所さんを探せ・やらないことリスト・夜も眠れない問題・トレードオフスライダー・技術的解決策・期間を見極める・何がどれだけ必要か

---

### 第9章：組織のジャーニーを続ける（組織のトランスフォーメーション②）

**核心命題**: 1980年代以来の日本組織は「深化」に最適化されており「探索」が欠落している。この呪縛を解くために「業務スクラム」を通じて探索のOSを通常業務に適用し、アジャイルの「縦糸（構造化）」と「横糸（コミュニティ）」で組織全体を編み直す。

**アジャイルハウスの3フロア**:
```
3F：組織のアジャイル化（縦糸＋横糸）
  ├─ 縦糸：マネジメントと現場のフィードバックループ（アジャイルディビジョン×ブリゲード）
  └─ 横糸：組織横断のアジャイルコミュニティ（ハンガーフライト）
2F：仕事のアジャイル化（業務スクラム）
  └─ プロダクト開発以外の通常業務にスクラムを適用
1F：チームのアジャイル化（スクラム実践）
```

**業務スクラムの適用対象**:
- プロジェクト型の仕事（明確な目標・期限あり）
- 定常型の業務（繰り返し性があり改善が求められる）
- 探索型の取り組み（目標があいまいで変化が大きい）

**「ものわかり」の習慣化**（コルブの経験学習を組織に実装）:
ふりかえりで行為を棚卸し → 効果的な工夫を見つける → 工夫が機能する背景・条件を整理 → 再利用可能なよう工夫に名前をつける

**ハンガーフライト**: 飛行機乗りが格納庫で経験を語り合った慣習に由来。組織横断の経験共有の場。外部事例をそのまま持ち込まず、文脈ごと共有し、パターン（型）として整理する。

**著者の最後のメッセージ**:
> 変革を始められるのは経営や一握りの戦略担当者だけではない。組織の一員であれば、誰でも始めることができる。自分たちのいる場所の、この先に希望を持てるようにしたいという意志を止めることは誰にもできない。変革は、あなたから始まる。

## 批判的視座

- **前提の偏り**: 本書の想定読者はIT・ソフトウェア開発を基盤とする組織。製造業・医療・公共など非IT産業でのDXには別の適応が必要であることが指摘不足。
- **規模の問題**: アジャイルブリゲード・ディビジョンの設計は数百人以上の大企業を前提としている。中小企業での適用可能性は読者が自ら判断しなければならない。
- **外部人材依存**: 「外部パートナー招聘」を多用するが、コスト・品質・依存リスクへの言及が少ない。「招聘」と「調達」の境界は精神論に依存しがち。
- **成果測定の曖昧さ**: DX推進指標・DXクライテリアが示されるが「どのスコアが合格か」の基準が組織任せで、判断が困難な組織には使いにくい。
- **アジャイル万能論の危険**: 「すべての仕事にアジャイルを」の方向性があるが、計画駆動が適切な局面（規制産業・安全クリティカル・大型インフラ）への目配りがやや弱い。

## 用語・前提の整理

- **ジャーニー**: 1〜3か月程度の時間単位（スプリントより大きく、プロジェクトより小さい）。始まりと終わりにふりかえり・むきなおりを行う
- **ふりかえり**: YWT（やったこと/わかったこと/次にやること）形式。過去を棚卸し現在の行動を正す
- **むきなおり**: 向かうべきゴール・目的を捉え直し、現在の取り組みを見直す。ふりかえりの上位概念
- **MVP（Minimum Viable Product）**: 想定利用者にとって価値があり、最小限の範囲に絞って作るプロダクト
- **セットベース**: 最初から1つに決め打ちせず、複数の選択肢（セット）を持ち検証によって絞る意思決定法
- **深化**: 既存事業を効率化・洗練化させるケイパビリティ
- **探索**: 新たな価値を見つけるための仮説検証・実験ケイパビリティ
- **CoE（Center of Excellence）**: DX推進における戦略と現場を繋ぐ専門チーム。アジャイルな運営が推奨される
- **アジャイルブリゲード**: 組織の水平分断を越境するための越境専門チーム。探索の専門性と希少技術の2本柱
- **アジャイルディビジョン**: 複数のアジャイルブリゲードを統括し、戦略のフロー効率性とリソース効率性を管理するメタチーム

## 実践チェックリスト

□ DXを「システム刷新」と定義せず、「組織変革の旗印」として社内で共通言語化する
□ 「なぜ変えるのか？」ではなく「なぜ変えないのか？」を問い直すカルチャーを作る
□ DXの4段階（業務デジタル化→スキルTFM→ビジネスTFM→組織TFM）でジャーニーマップを描く
□ 各ジャーニーの開始と終了を明示し、ふりかえり・むきなおりを実施する
□ チャット（Slack等）・Web会議・クラウドファイルの三点セットを最優先で導入する
□ 組織ポリシー変更に向けて「正攻法・外部代行・出島」の3作戦から現状に合うものを選ぶ
□ 協働の型（見える化・場作り・一緒にやる）をツール導入と同時に展開する
□ タスクマネジメントの基本（ゴールデンサークル→タスク洗い出し→1〜2週間サイクル）をチームに定着させる
□ DXクライテリアや星取表でチームのケイパビリティを可視化してから研修計画を立てる
□ 仮説検証では10名1セットのインタビューを最初のサイクルとして設計する
□ 仮説キャンバスで「わかっていること」と「わかっていないこと（仮定）」を必ず分類する
□ PSfit（問題と解決策の一致）確認前にMVPの大規模開発に着手しない
□ ターンアラウンドを意識し、「検証の準備・実施・学習修正」の一周を1か月以内にする
□ CoEをDX推進の垂直分断解消のために早期設置し、スクラムで運営する
□ アジャイルブリゲードを立ち上げる前に、立ち上げ部署内で小さな実績を作っておく
□ 「ブリッツトラスト」を意識し、越境先との最初の3か月は集中投資で信頼を獲得する
□ アジャイルディビジョンでフロー効率性（優先度）とリソース効率性（人材配置）を同時に管理する
□ 業務スクラムを通じてプロダクト開発以外の通常業務にも探索のOSを適用する
□ ハンガーフライト（経験共有の場）を組織横断で定期開催する
□ 「ものわかり」習慣として、効果的な工夫に名前をつけ再利用可能な型として整理する

## よくある落とし穴

1. **「屏風のトラ」DX**: 立派な戦略文書はあるが実行の算段がない。「絵は自分たちで描く」必要がある
2. **ビッグフロントのDX**: 最初から大規模な施策（全社アジャイル研修・アクセラレーター）を打ち上げ、「お祭り」で終わる
3. **ツール≠変革の錯誤**: Slackを入れても協働の型がなければチャットでメールをやるだけ。ツールと協働カルチャーはセットで導入する
4. **「足元がお留守」の罠**: 仮説検証・アジャイルなど新しいスキルに目を奪われ、基本的なチームタスク管理が機能していない
5. **PDCA廃止の誤り**: 「PDCAは古い、OODAが正しい」という二項対立に陥る。タイムボックスとバランスで調整するものであって廃棄するものではない
6. **インタビュー数の誤り**: 3〜5名で判断してしまう。10名1セットを基本に、顕著な結果が出るまで繰り返す
7. **MVPの肥大化**: 「最小限」と言いながら機能を詰め込みすぎる。MVPはターンアラウンド最短化の手段と再定義する
8. **垂直分断の放置**: 事業開発が進むほど現場と経営のズレが拡大する。CoEを後から設置しようとすると間に合わない
9. **外部人材を「調達」する思想**: パートナーを組織変革のリスクを共に負う「招聘」相手として捉えない
10. **出島に閉じ続ける**: 出島の成果が本土に伝播しなければ組織変革にならない。越境のタイミングと方法を最初から構想しておく
