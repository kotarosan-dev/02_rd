## <a id="core"></a>コアフレームワーク：本書の骨格

### 衝撃の数字：日本企業IT投資の実態

- **工数比較**：日本企業は欧米比で **3倍以上** のIT投資工数がかかる
- **DX成功率**：BCGの2020年調査で、日本企業のDX成功率は **1.4%**（世界平均30%）
- **ERPカスタマイズ率**：日本95%超、米国40%以下
- **ERP導入期間**：日本3〜5年、海外1〜2年
- **「2025年の崖」経済損失**：年間最大 **12兆円**（経産省推計）
- **IT人材の偏り**：日本のIT専門人材の **74%がITベンダー所属**、米国は逆転（65%が事業会社）
- **CIO設置率**：日本15.9%、CDO：16.6%（欧米主要企業は大半が設置）
- **ベンダー単価上昇**：2022年時点で1人あたり約100万円／月、5年前比20%以上上昇

### 失敗の根本構造：「オペレーションドリブン」vs「データドリブン」

```
日本企業の罠（オペレーションドリブン）
  ↓
現行業務プロセスを最優先し、システムをカスタマイズし続ける
  ↓
ERPや基幹システムが「オリジナルの怪物」へ変貌
  ↓
保守コストが膨張、新機能追加が困難、人材が育たない
  ↓
さらにカスタマイズで対応…（負のスパイラル）

目指すべき転換（データドリブン）
  ↓
標準パッケージの業務プロセスに社内業務を合わせる
  ↓
データを資産として設計し、全社で共有・活用
  ↓
改善サイクルが速くなり、AI活用の基盤が整う
```

### IT投資の4つのマインド（本書の中核フレーム）

| マインド | 内容 | 陥りやすい罠 |
|---|---|---|
| ① 戦略連動 | IT投資をビジネス戦略に直結させる | IT部門任せ、技術先行 |
| ② アウトカム志向 | 稼働ではなく成果で評価する | KPIが「稼働率」「予算消化」になる |
| ③ アジャイル思考 | 完成を待たず小さく試して改善する | 要件固めてからという呪縛 |
| ④ データ資産化 | データをビジネスの燃料と捉える | システムがデータを「閉じ込める」設計 |

### IT・DX戦略の3視点（失敗診断の座標軸）

```
┌──────────────────────────────────────┐
│  1. 戦略・ロードマップ                │  ← 「なぜ・何を・いつまでに」
│     全社DX戦略の策定・優先順位・時間軸  │
├──────────────────────────────────────┤
│  2. ガバナンス・プロセス              │  ← 「誰が決め・どう管理するか」
│     意思決定体制・PMO・ベンダー管理    │
├──────────────────────────────────────┤
│  3. 組織・体制                        │  ← 「誰が実行するか」
│     CIO/CDO/CHRO・内製化・人材育成    │
└──────────────────────────────────────┘
```

## <a id="intro"></a>はじめに：問題提起

コロナ禍以降、日本企業のデジタル化遅延が露わになった。BCGのコンサルタントたちが日常的に目撃する光景は共通している——多大なコストをかけ、長期間をかけて開発したシステムが、稼働直前に凍結されるか、稼働後も「使われないシステム」になる。

本書の出発点となる問いは明快だ：「なぜ日本企業だけがこんなに苦しむのか？」  

答えは技術の問題ではなく、**意思決定の構造と組織文化の問題**にある。本書はBCGが日本企業数十社と向き合ってきたプロジェクトの失敗事例と成功事例をもとに、失敗のパターンを体系化し、処方箋を提示するものだ。読者対象は「経営者・IT部門長・DXを推進したいビジネスパーソン」。

キーワードとして「失敗の本質」が援用される。日本軍の組織論的研究を引き、「空気」による意思決定——客観的根拠ではなく場の雰囲気で重大な判断がなされる構造が、IT投資でも再現されていることを指摘する。

## <a id="ch1"></a>第1章：なぜ日本企業はIT投資で失敗するのか

### 業界別IT投資の現状

**小売業**：ファーストリテイリングはRFIDタグによる全商品管理とAI需要予測で、在庫圧縮と欠品率低減を同時達成。一方、多くの国内小売は「POS導入が精いっぱい」の状態が続く。  
**飲食業**：すかいらーくは注文・配膳ロボットと厨房自動化を連動。スターバックスはモバイルオーダーで店舗動線を再設計した。ここでも先進事例と停滞事例の差は「IT投資をコストと見るか、戦略資源と見るか」に帰結する。  
**金融・保険**：地方銀行は1990年代設計のレガシーシステムを抱え、クラウド移行さえ困難。大手保険は「100億円を投じたプロジェクトを途中停止した」事例が複数存在し、BCGが実際に目撃している。  
**製造業**：グローバル展開において「翻訳力」の欠如が致命傷になる。IT部門がビジネス要件を技術仕様に変換できず、ベンダーとの対話が成立しない。

### 「2025年の崖」の正体

経済産業省が2018年の「DXレポート」で警告した「2025年の崖」は、次の構造から生じる：
1. 多くの基幹システムが **2025年前後でサポート終了**（ERPやOS）
2. 維持・保守にかかるIT予算が **80%超**を占め、新規投資できない
3. 保守できるIT人材が **定年退職ピーク**を迎える
4. 移行しない場合、年間最大12兆円の経済損失が生じる

この「崖」を前に、多くの企業は「とにかく移行だけ」を急ぐ。しかし移行を「現行踏襲」で行うと、同じ構造的問題を引き継いだ新システムができあがる。

### IT人材問題の深刻さ

> 「ベンダーに任せきりで要件定義が曖昧なまま着工し、結果的に追加開発が雪だるま式に増えた。数十億円を投じても期限に間に合わず失敗扱いだ」（大手流通業）

この証言が典型的な構造を示す。日本のIT人材は約140万人だが、そのうちの **70%がITベンダー企業に所属**する。事業会社側にIT能力がないため、要件定義すらベンダー任せになり、「ベンダーが提案した仕様を丸のみする」スパイラルが固定化する。

ベンダー市場では人材不足が深刻化し、単価は2022年時点で月100万円程度まで上昇（5年前比20%超）。プロジェクトの長期化・追加開発の積み重なりが、コスト爆発をさらに加速させる。

### なぜカスタマイズをやめられないのか

日本企業のERPカスタマイズ率95%超という数字の背景には、根深い組織論的理由がある。

- **「うちの会社は特殊だ」という神話**：業務部門が標準プロセスへの適応を拒否する
- **稟議・根回し文化**：現場の同意なしには変更できない意思決定構造
- **担当者の個人知識**：Excelマクロや独自帳票が属人的に蓄積され、システムにフィットしない
- **IT部門の弱さ**：ビジネス側に「NO」を言えず、カスタマイズ要求を全て受け入れる

## <a id="ch2"></a>第2章：IT・デジタルプロジェクト成功に向けての準備

### 経営者が持つべき「デジタルリテラシー」の基礎

本書が強調するのは、経営者がITの詳細を理解する必要はないが、「何を問うべきか」を知ることが必須だという点だ。問うべき問いは5つある：

1. このIT投資は、どのビジネス戦略目標に紐づいているか？
2. ROIの前提条件は何か、そして誰がそれを追跡するのか？
3. 失敗したとき、どの段階でどう判断するのか？
4. 内製とベンダーの役割分担は適切か？
5. データは誰のものか、どう活用するのか？

### 4つのマインドの深堀り

**戦略連動**：IT投資の意思決定が「IT部門の内部論理」ではなく「全社戦略の優先順位」から行われる状態。典型的な失敗は「技術者がかっこいいと思う技術を選ぶ」プロジェクト。

**アウトカム志向**：「システムが稼働した」ではなく「売上が上がった」「コストが下がった」「顧客体験が改善した」で評価する。KPI設計の段階から成果指標を定義する。

**アジャイル思考**：完全な要件定義を待たず、最小限の機能（MVP）を早期にリリースし、ユーザーフィードバックをもとに改善する。日本企業は「全て仕様を固めてからでないと開発できない」という誤解に縛られやすい。

**データ資産化**：システムをデータの「通路」ではなく「資産の器」として設計する。どのデータをどの粒度で蓄積し、どう活用するかを、システム設計より先に決める。

### 組織体制：CIO・CDO・CHROの三位一体

| 役職 | 役割 | 日本の現状 |
|---|---|---|
| CIO（最高情報責任者） | IT投資の全体最適・ガバナンス | 設置率15.9%、多くはIT部門長の兼任 |
| CDO（最高デジタル責任者） | DX戦略の牽引・データ活用 | 設置率16.6%、権限が曖昧な場合が多い |
| CHRO（最高人事責任者） | IT人材の採用・育成・組織変革 | CIOやCDOと連携できていない場合が多数 |

3者が連携しないと、「技術は変えたが人が追いつかない」「人材は採ったが戦略と合っていない」「戦略は決まったが組織が動かない」という分断が起きる。

### 内製化戦略

システム開発内製化率：日本全体で20%超、1000人超の大企業で40%程度。欧米主要企業では50〜80%が内製化を志向している。内製化のメリットは「スピード」「ナレッジの蓄積」「コスト構造の変換」だが、日本でのハードルは：

- **採用難**：高スキルエンジニアは年収1,000〜2,000万円を求めることも多く、既存の給与体系とギャップ
- **文化的摩擦**：エンジニア文化と既存の日本型雇用制度の衝突
- **中途半端な内製化**：ベンダー依存を解消せず「追加のコスト」になるケース

## <a id="ch3"></a>第3章：プロジェクトの各フェーズでやるべきこと

### 3フェーズの全体像

```
フェーズ1：企画・構想
  目的：「なぜやるか」「何を変えるか」を経営レベルで合意する
  成果物：IT戦略、対象スコープ、投資規模の大枠、ROI仮説

フェーズ2：要件定義
  目的：「何をシステムが行うか」を精緻化し、ベンダーへの発注条件にする
  成果物：業務要件書、システム要件書、RFP、選定基準

フェーズ3：実装・導入
  目的：システムを構築し、業務に組み込む
  成果物：稼働システム、移行完了、ユーザートレーニング
```

### フェーズ1の罠：企画の段階で失敗している

「プロジェクトの失敗の70%は要件定義以前に原因がある」という本書の主張は重い。企画段階でよく起きる問題：

- **目的の曖昧さ**：「DXをやる」「クラウド化する」が目的化し、ビジネス成果が定義されない
- **スコープのクリープ**：経営者の一言で範囲が拡大し続ける
- **ROI算定の甘さ**：楽観的な想定で投資承認を得るが、実現のロジックが組まれていない
- **ステークホルダーの合意不足**：IT部門だけが動き、事業部門が「他人事」

対策：**「AS-IS / TO-BE マッピング」と「価値仮説の文書化」**を企画フェーズの必須成果物とする。

### フェーズ2の罠：要件定義の失敗パターン

要件定義での典型的失敗：

- **現行業務の写し**：AS-ISの業務を要件にそのまま落とし込み、TO-BEの設計が行われない
- **ユーザー不在の要件**：IT部門とベンダーだけで決め、実際のユーザーが関与しない
- **「あとで調整」の先送り**：難しい判断を「実装フェーズで」と先送りし、工数爆発の原因になる
- **RFPの質の低さ**：評価軸が「価格」「実績」のみで、技術的適合性・協業スタイルが評価されない

対策：要件定義に **業務部門のキーパーソン（現場マネジャー）を正式メンバーとして参加させる**。IT部門やベンダーではなく、ユーザー代表が「これで業務が変わる」と言えるまで確認する。

### フェーズ3の罠：実装フェーズで積み増す問題

> 「仕様変更禁止のルールを設けたが、経営から『やはりこの機能が必要だ』と言われ、例外対応を繰り返した。最終的に当初見積もりの2.5倍の工数になった」

実装フェーズの主要な失敗パターン：

- **スコープクリープの再発**：要件定義で決めたはずのスコープが変動し続ける
- **テスト工程の軽視**：コスト圧縮でテストが削られ、本番稼働後に致命的バグが露見
- **移行リスクの軽視**：旧システムから新システムへのデータ移行に失敗し、業務が止まる
- **変更管理の欠如**：ユーザーへのトレーニングと業務変革を「システム稼働後」に後回しにする

対策：**ゲートレビュー制度**の導入。各フェーズの移行前に経営が確認する「Go/No-go判断」を制度化する。

## <a id="ch4"></a>第4章：失敗から学び失敗を活用する

### 失敗の種類と対応の違い

本書は失敗を3タイプに分類する：

| タイプ | 特徴 | 対処法 |
|---|---|---|
| 回復可能な失敗 | 早期に検知でき、軌道修正が利く | 迅速な判断・ピボット |
| 高コストな失敗 | 損害は大きいが学習価値がある | 徹底的な原因分析と共有 |
| 壊滅的な失敗 | 事業継続に影響する | 予防設計・リスク管理の強化 |

多くの日本企業は「失敗を隠す」文化があり、回復可能な段階で止められたはずの失敗が壊滅的な失敗まで拡大する。

### 軌道修正のタイミングを逃さない

「プロジェクトを止める」判断が最も難しい。**埋没コストバイアス**（「ここまでかけたお金が無駄になる」）と、**組織の空気**（「ここで止めた人間が戦犯になる」）が、明らかに失敗しているプロジェクトを走り続けさせる。

本書が提示する判断基準：
- **コストが当初見積もりの150%を超えたら**：プロジェクト全体の見直しを必須とする
- **スケジュールが6ヶ月以上遅延したら**：ROI仮説を再検証する
- **ビジネス環境が大きく変化したら**：計画の前提を再点検する

### 失敗ナレッジの組織的活用

失敗事例を「不名誉な事件」として封印するか、「学習の財産」として共有するかが、組織の次のプロジェクト品質を決定する。具体的な制度として：

- **プロジェクト事後分析（Post-mortem）の義務化**：成功・失敗を問わず実施
- **失敗事例データベースの構築**：次のプロジェクト企画時の参照情報として活用
- **「失敗を語れる」心理的安全性の醸成**：ドイツや米国の先進企業が実践する文化設計

## <a id="ch5"></a>第5章：日本企業が勝つためのアプローチ

### スタンダードデータへの転換

日本企業がグローバル競争で遅れを取る最大要因の一つが「データの非標準化」だ。各部門・各システムが独自のデータ形式を持ち、横断的な分析が困難になる。

**スタンダードデータの設計原則**：
1. **マスターデータ管理（MDM）**：顧客・商品・組織などのコアデータを1か所で一元管理
2. **データモデルの標準化**：業界標準（IFRS対応、EDI標準）への準拠を前提とする
3. **APIによる連携**：部門間のデータ連携をAPIで明示化し、密結合（スパゲッティ連携）を解消

### 業界連携・コンソーシアム戦略

競合企業が同じIT基盤（共通プラットフォーム）を使うことで、個社のシステム維持コストを下げ、差別化に集中する発想が欧米では定着している。日本での事例はまだ限定的だが、金融業界での勘定系共有や、小売業でのEDI標準化に芽がある。

「競争と協力の軸」を正確に見極め、ITインフラは協調し、顧客体験と商品力で差別化する戦略が重要だ。

### 生成AIの活用と現実

BCGが実施した大手保険会社での生成AI活用事例：
- **開発工数50%低減**：コード生成・レビューへの生成AI適用
- **欠陥30%低減**：テストケース生成とバグ予測への適用

ただし本書は楽観的な過大評価を戒める。生成AIは「業務担当者の仕事をなくす」のではなく、「より高度な判断業務に集中できるようにする」ツールだ。AIを使いこなすためには、データ品質の担保・プロンプト設計のスキル・AI出力を検証できるドメイン知識が不可欠で、これらは一夜にして得られない。

### IT人材の育成と採用

国内IT人材は **約140万人** だが、欧米との差は量より構造にある。事業会社にIT人材が内製化されていない日本では、「ITを語れる経営者」「技術を理解しているビジネス担当者」の絶対数が圧倒的に少ない。

提言する人材像：
- **デジタルゼネラリスト**：技術の詳細は知らないが、ITとビジネスを橋渡しできる人材
- **DXプロデューサー**：プロジェクトを横断し、関係者を調整・推進できるリーダー
- **データサイエンティスト**：データから意味を引き出し、意思決定に接続できる専門家

### 国・行政の役割

日本全体としてのDX遅延は、個社の問題だけでなく、**行政のデジタル化遅延**と連動している。紙・ハンコ文化が規制で維持される限り、民間企業のDXには自ずと上限がある。マイナンバー活用、デジタルID整備、規制のデジタル対応（レグテック）など、公共インフラとしてのデジタル基盤整備が急務だ。

## <a id="outro"></a>おわりに：AIと人材への警鐘

著者たちが最後に示す問いは厳しい。生成AIが急速に普及する中で、「IT人材が不要になる」という楽観論と、「AIを使いこなせない組織は淘汰される」という危機論の両方が台頭している。

本書の立場は明確だ——**技術は変わるが、変革を主導する人間の役割は増す**。AIに仕事を奪われるのではなく、AIを正しく使えない経営者・組織が競争で脒れる。「失敗の本質」は技術にあるのではなく、技術を生かせない意思決定と組織文化にある。

その意味で、本書が処方箋として提示するものは最終的に「人の問題」への答えだ：経営者が正しい問いを持ち、組織が学習し続け、失敗を資産に変えられるか。

## 原文引用（2本）

> 「ベンダーに任せきりで要件定義が曖昧なまま着工し、結果的に追加開発が雪だるま式に増えた。数十億円を投じても期限に間に合わず失敗扱いだ」  
> — 大手流通業（匿名）

この証言は第1章で示される典型的失敗パターンを象徴する。「ベンダー任せ」「曖昧な要件」「追加開発の積み上げ」という三要素が揃ったとき、プロジェクトは自走する暴走列車と化す。

> 「空気」による意思決定——客観的根拠ではなく場の雰囲気で重大な判断がなされる構造は、IT投資でも再現されている。  
> — 本書（「失敗の本質 日本軍の組織論的研究」への言及箇所より）

本書が「失敗の本質」という古典を援用するのは偶然ではない。70年以上前に日本軍が陥った組織的失敗の構造が、現代のIT投資にそのまま再現されているという著者の診断は、単なるレトリックではなく、日本企業への根本的な問いかけだ。

## 批判的視座

本書が提示するフレームワークは実践的だが、以下の点には留意が必要だ：

**限界1：コンサルティング視点の偏り**  
BCGの提言は大企業向けに設計されている。中小企業でCIO/CDOを置くことは現実的でなく、「スタンダードデータへの転換」も中小では過重な投資になりかねない。

**限界2：成功事例の選択バイアス**  
ファーストリテイリングやスターバックスの事例は「圧倒的なリソースを持つ企業」の成功であり、平均的な日本企業に適用できるかどうかは別論だ。

**限界3：変化のスピードへの過信**  
「標準プロセスに業務を合わせる」という処方箋は正しいが、業務部門の抵抗と組織慣性を考慮すると、現実には5〜10年単位の変革が必要なケースも多い。本書の提言はやや短期楽観的だ。

## <a id="checklist"></a>実践チェックリスト（20項目）

### 経営者向け

- [ ] 現在のIT投資が、どの戦略目標に紐づいているか即答できるか
- [ ] IT投資の成果をROI（ビジネス成果）で評価する仕組みが社内にあるか
- [ ] プロジェクトの失敗判断基準（コスト・スケジュールの閾値）が明文化されているか
- [ ] CIO・CDO・CHROの役割が明確に定義されているか（設置しているかどうかではなく）
- [ ] 「失敗を語れる」心理的安全性が自分の組織にあると確信を持って言えるか

### IT責任者・プロジェクトリーダー向け

- [ ] 現在進行中のプロジェクトの要件定義に、業務部門のキーパーソンが正式参加しているか
- [ ] ゲートレビューの判断基準と責任者が、プロジェクト開始前から決まっているか
- [ ] ベンダーへのRFPに、価格・実績以外の評価軸（技術適合性・協業スタイル等）が含まれているか
- [ ] 現在のIT予算の中で「維持・保守」と「新規投資」の比率を把握しているか
- [ ] データ（顧客・商品・組織）のマスターデータ管理（MDM）の設計があるか
- [ ] プロジェクト完了後に必ずPost-mortemを実施する制度があるか
- [ ] 現行業務プロセスをERPやパッケージ標準に合わせる意思決定権限が誰にあるか明確か

### 事業部門・ビジネス担当者向け

- [ ] IT部門とのコミュニケーションで「うちは特殊だ」を主な理由にカスタマイズを求めていないか
- [ ] 新システム導入時に、自分の業務がどう変わるかを理解し、変革を受け入れる準備があるか
- [ ] 日常業務で使うデータが「どこから来て、どこへ行くか」を説明できるか

### 全社共通

- [ ] 社内のIT人材（内製エンジニア・データサイエンティスト）の現状数と目標数が定義されているか
- [ ] 生成AIの活用ポリシーが組織として策定されているか（禁止でも全面採用でも、明確な方針があるか）
- [ ] 競合他社のIT投資動向と自社の差分を年次で評価しているか
- [ ] 現在の基幹システムのサポート終了時期（2025年の崖に該当するか）を確認しているか
- [ ] 失敗したプロジェクトの事例を、組織的に記録・共有・参照する仕組みがあるか

## よくある落とし穴（ワーニングリスト）

1. **「クラウド化すればDXだ」**：クラウドは手段であり、ビジネスモデルが変わらなければDXではない
2. **「AIを導入したらIT部門は不要」**：AI管理・データ品質管理・倫理的判断のためにIT人材の重要性は増す
3. **「標準パッケージに合わせると競争力が落ちる」**：競争優位はITではなく顧客体験・商品・サービスで作る
4. **「プロジェクトが遅れているが、止めるに止められない」**：埋没コストバイアス。150%超えたら強制ゲートレビュー
5. **「IT部門に任せておけば大丈夫」**：経営者の関与なしにDXは成功しない——これが本書の大前提
