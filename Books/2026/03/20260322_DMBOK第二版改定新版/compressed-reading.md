## 読書ナビゲーション

この本は「データをどう管理するか」を個別技術の寄せ集めとしてではなく、企業経営・業務・IT・ガバナンス・文化変革をまたぐ一つの体系として示すリファレンスである。中核メッセージは一貫していて、データは単なるITの副産物ではなく、価値とリスクを同時に持つ企業資産であり、そのため管理対象はデータベースだけでなく、定義、品質、責任、標準、組織、変革まで含まれる。

読み方のコツは、17章を均等に読むことではない。まず第1章で全体原則をつかみ、次に第3章・第12章・第13章でガバナンス、メタデータ、品質の土台を押さえる。そのうえで、自分の関心に応じて第4〜14章の各知識領域へ降りていくと理解が速い。最後に第15〜17章を読むと、「なぜ分かっていても実装できないのか」と「どう定着させるか」が見える。

## 概念マップ

### 1. 本書の中核命題

本書の骨格は次の一文に集約できる。

> **「データは企業にとって重要な資産である。」**（第1章）

ただし本書が言う「資産」は、単に売れるとか貯められるという意味ではない。データは業務遂行、意思決定、分析、自動化、規制遵守、顧客体験の土台である一方、低品質・漏洩・誤用・不整合の形でコストと損害も生む。したがって、価値化とリスク管理を同時に回す統合的な管理が必要になる。

### 2. 何を管理する本なのか

本書が扱う対象は広いが、整理すると以下の5層に分けられる。

| 層 | 何を扱うか | 典型テーマ |
|---|---|---|
| 原則層 | データをどう捉えるか | 資産性、ライフサイクル、全社視点、業務主導 |
| 統制層 | どう決め、どう守らせるか | ガバナンス、ポリシー、標準、スチュワード制 |
| 設計層 | どう構造化するか | アーキテクチャ、モデリング、メタデータ |
| 実行層 | どう動かし守るか | ストレージ、セキュリティ、統合、品質運用 |
| 活用層 | どう価値化するか | MDM、DW/BI、データサイエンス、組織変革 |

### 3. DMBOKフレームワークの読み解き

本書ではDAMAホイールを使って知識領域を説明するが、実務上は次の理解が分かりやすい。

| 役割 | 知識領域 |
|---|---|
| 土台を整える | ガバナンス、メタデータ、品質 |
| 設計する | アーキテクチャ、モデリング |
| 守って回す | ストレージ、セキュリティ、統合 |
| 共通資産化する | 参照データ、マスターデータ |
| 活用する | DW/BI、ビッグデータ、分析 |
| 定着させる | 成熟度、組織、変革 |

## 判断基準表

| 状況 | 選択肢A | 選択肢B | 判断基準 |
|---|---|---|---|
| データ管理の起点 | IT主導 | 業務主導 | 本書は一貫して業務主導を推す。ITは手段であり、要件の起点ではない。 |
| 管理単位 | 部門最適 | 全社最適 | 局所最適でも始められるが、最終的には全社的視点へ接続できる設計が必要。 |
| データ品質対応 | 後工程で修正 | 発生源で予防 | 一時しのぎのクレンジングより、根本原因の除去が優先。 |
| 統合方式 | ポイントツーポイント | ハブ/標準インターフェース | 接続先が増えるほど後者が有利。複雑性と保守コストを抑えられる。 |
| MDM導入 | 一気に全領域 | 優先領域から段階導入 | 重要度が高く、比較的単純な領域から着手するのが安全。 |
| BI設計 | ツール起点 | ビジネス問い起点 | 先に問い、利用者、判断行動を定義し、その後ツールを選ぶ。 |
| ビッグデータ活用 | 全部集める | 目的に沿って収集・統制 | 収集自体は価値ではない。メタデータと用途設計がないと「データスワンプ」化する。 |
| 組織変革 | 組織図変更で完了 | 行動変容まで追う | 本書は後者。制度変更だけでは不十分で、文化に定着するまでが仕事。 |

## パターン/アンチパターン

| パターン（推奨） | アンチパターン（回避） | 理由 |
|---|---|---|
| データを資産として扱う | データをアプリの副産物とみなす | 価値測定、責任、投資判断ができなくなる。 |
| メタデータを管理の起点にする | 定義の曖昧さを放置する | 統合、品質、検索、説明責任の全てが崩れる。 |
| データ品質を業務ルールで定義する | 「正しいデータが必要」とだけ言う | 測定不能な要求は改善につながらない。 |
| 共有データにガバナンスをかける | 参照データやマスターデータを部門ごとに保持する | 不整合、重複、統合コストを増やす。 |
| 重要データから段階的に改善する | 全データを一気に高品質化しようとする | 実行不能になりやすい。優先順位が不可欠。 |
| 変革の危機感とビジョンを作る | ツール導入を変革そのものと誤認する | 行動変容が起きず、定着しない。 |

## 章別エッセンス

### 第1章 データマネジメント

本書全体の憲法。データは資産であり、価値、品質、メタデータ、ライフサイクル、リスク、リーダーシップを要件とするという原理を定める。特に重要なのは「データ管理は品質管理である」「データ管理にはメタデータが必要」「データ要件がITを決める」で、この3つを外すと後続章がすべて局所技術論に崩れる。

### 第2章 データ取扱倫理

データを扱えることと、扱ってよいことは別だと明言する章。倫理は規制遵守の上位概念であり、プライバシー、バイアス、虚偽的可視化、説明責任まで視野に入れる。本書が単なるITハンドブックではなく、データの社会的正当性まで扱う理由がここにある。

### 第3章 データガバナンス

データマネジメント全体を「正しく進めるための決定権体系」を扱う。戦略、ポリシー、標準、役割、課題管理、規制対応、変更管理の基盤。重要なのは、ガバナンスは管理そのものではなく、管理が正しく行われるようにする統制であるという区別。

### 第4章 データアーキテクチャ

企業全体のデータ青写真を定義する章。データモデルだけでなく、データフロー、依存関係、ロードマップまで含める。設計対象は「今あるシステム」ではなく、「組織が必要とするデータの全体像」。局所プロジェクトの都合で構造を決めないための視点を与える。

### 第5章 データモデリングとデザイン

エンティティ、属性、関係、キー、ドメイン、正規化、各種スキーマなど、実務でデータを記述するための文法書。モデルの目的はDB設計だけではなく、共通語彙と意思決定の共有基盤を作ることにある。概念・論理・物理の3層を混同しないことが重要。

### 第6章 データストレージとオペレーション

データベース運用、可用性、性能、バックアップ、リカバリ、保持、移行などを扱う。強調されるのは、運用は単なる保守ではなく事業継続の中核だということ。構成管理、環境分離、リカバリ計画、SLAが重視される。

### 第7章 データセキュリティ

機密性、規制対象、認証、権限付与、監査、暗号化、マスキング、侵入対策など、データを守る章。特に有用なのは「機密性レベル」と「規制カテゴリ」を分けて考える点で、社内都合の制限と法的制限を混同しない判断軸になる。

### 第8章 データ統合と相互運用性

ETL/ELT、レイテンシ、連携モデル、ESB、SOA、イベント処理など、データを動かす章。メッセージと変換ルールにガバナンスが必要だと繰り返し説く。単純接続を増やすより、標準モデルやハブで複雑性を抑えるのが王道。

### 第9章 ドキュメントとコンテンツ管理

非構造化データ、レコード管理、タクソノミ、eディスカバリ、ECM を扱う。ここでの核心は「構造化されていないから管理不要」ではなく、むしろメタデータ、分類、保持、監査が必要になるという点。データベース外の情報資産も統治対象になる。

### 第10章 参照データとマスターデータ

共有データを共通資産化する章。参照データは意味の基準、マスターデータは重要エンティティの信頼できる表現。MDM はツールではなく、人・ルール・統合・スチュワード制を含んだ運用体系であり、段階導入が原則。

### 第11章 データウェアハウジングとビジネスインテリジェンス

統合データの蓄積と意思決定支援の章。Inmon系とKimball系の違いを踏まえつつ、本質は「分析しやすい形に統合すること」にある。ユーザー、レポート、マート、キューブ、リリース管理まで扱い、BI を製品として運営する視点を与える。

### 第12章 メタデータ管理

本書の隠れた中核。用語集、データ辞書、リネージ、構成、サービス、品質情報まで含めたメタデータを統合・管理する。メタデータがなければデータ資産は検索も説明も統制もできない。多くの組織で最初の改善起点になりうる章。

### 第13章 データ品質管理

品質を「目的適合性」と定義し、重要データ、評価軸、業務ルール、初期アセスメント、根本原因分析、運用化まで一連で説明する。品質問題の多くは入力ミスより、ガバナンス不足やプロセス設計不備から生まれるという指摘が実務的に重い。

### 第14章 ビッグデータとデータサイエンス

ビッグデータ、機械学習、予測分析、視覚化、データレイクを扱う。新技術礼賛ではなく、メタデータ、品質、セキュリティ、ガバナンスがなければ機能しないと位置づけるのが本書らしい。データレイクが「スワンプ」化する警告は重要。

### 第15章 データマネジメント成熟度アセスメント

データマネジメント能力を評価し改善ロードマップに落とす章。何が足りないかを共通言語で可視化し、投資と変革の優先順位を決めるための道具になる。単なる診断で終わらず、再評価を前提にしている点が実務的。

### 第16章 データマネジメント組織と役割期待

地方分権型、中央集権型、ハイブリッド、連邦型などの組織モデルを比較し、役割と責任を明確にする章。重要なのは、完璧な組織図はなく、文化・成熟度・事業構造に合わせて進化させるべきだという立場。

### 第17章 データマネジメントと組織の変革

本書の締めくくり。制度や技術を入れただけではデータマネジメントは根付かない。危機感、変革推進チーム、ビジョン、短期成果、文化定着というチェンジマネジメントの原則を通じて、なぜ多くのデータ施策が途中で止まるのかを説明する。

## 落とし穴マップ

| 落とし穴 | 症状 | 回避策 |
|---|---|---|
| ツール先行 | 導入したのに意思決定が良くならない | 業務課題、判断、責任、品質要件を先に定義する |
| 用語未整備 | 部門ごとに同じ言葉の意味が違う | 用語集、メタデータ、スチュワード制を先に置く |
| 品質後回し | 下流で手戻りと修正が常態化する | 発生源対策、ルール化、継続測定に切り替える |
| MDMを製品視する | ツール導入後も重複や不整合が残る | ガバナンス、マッチング方針、保守責任を設計する |
| BIを報告機能だけと捉える | 利用が広がらず、現場判断に接続しない | 利用者別の配信戦略とセルフサービス設計を行う |
| 変革を軽視する | 最初だけ盛り上がって定着しない | 危機感、ビジョン、短期成果、反復コミュニケーションを設計する |

## 原著への架け橋

この圧縮読本で残したのは、全体体系、章ごとの役割、実務判断の基準である。 ただし原著が持つ本当の価値は、各章の細かい用語定義、図表、評価尺度例、具体的な管理項目、そして「なぜその知識領域が他領域と結びつくのか」という網目構造にある。 特に自分の現場で制度設計や標準化を行う人は、第3章・第12章・第13章・第15章・第17章を原著で丁寧に読む価値が高い。 逆に実務で今すぐ使う観点だけなら、この圧縮版と `structure-map.md` で全体像は十分掴める。
