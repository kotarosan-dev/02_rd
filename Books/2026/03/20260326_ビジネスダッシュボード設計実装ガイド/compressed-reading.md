## 読書ナビゲーション

本書は、BIツールの操作説明書ではありません。中心にあるのは、**「使われるダッシュボード」をどう設計し、どう定着させるか**という方法論です。著者たちは、ダッシュボードを単なる可視化画面ではなく、`ビジネス課題の把握 -> 意思決定 -> アクション` を支える業務装置として扱います。そのため、重要なのは「どんなチャートを置くか」より先に、`誰のどんな意思決定を支えるのか` を定義することです。

読み方のコツは、各章を個別の知識として覚えるのではなく、**一つの連続した設計フロー**として掴むことです。  
`要求定義` で目的を定め、`要件定義` で利用シーンを具体化し、`分析設計` で問いと指標を定義し、`デザイン` で認知負荷を下げ、`データ準備` で実装可能な粒度へ落とし、`運用` で使われ続ける仕組みにする。  
この順序が本書の価値です。

## コアフレームワーク

### 1. 使われるダッシュボードの定義

| 要素 | 意味 | 実務上の問い |
|---|---|---|
| 目的に直結する | 何のビジネス課題のための画面かが明確 | このダッシュボードは何を改善するためのものか |
| 意思決定につながる | 見たあとに行動が変わる | これを見た人は何を判断し、何を実行するのか |
| 継続的に見られる | 更新され、速く、使い続けられる | 毎日/毎週の業務に組み込めるか |

本書が最初から最後まで守っているのは、**「可視化して終わり」にしない**という原則です。

### 2. ダッシュボード構築の全体フロー

| フェーズ | 目的 | 主成果物 |
|---|---|---|
| 要求定義 | ビジネス課題と目標を定める | KGI/KPI/CSF、課題整理 |
| 要件定義 | 誰がどう使うかを定める | 想定ユースケース、全体構成 |
| 分析設計 | 問い・指標・比較軸を定める | ダッシュボード詳細設計書 |
| デザイン | 見やすさと分析導線を作る | ワイヤーフレーム、モックアップ |
| データ準備 | 実現可能なデータ構造を作る | マートテーブル、更新ルール |
| 構築 | BI上で実装し整合性を取る | ダッシュボード本体 |
| 運用 | 利用定着・改善・保守を回す | レビュー、説明資料、改善運用 |

### 3. 要求定義フレーム

要求定義は「何を作るか」ではなく、**何の課題に効くかを明確にする工程**です。

| ステップ | 役割 | 具体内容 |
|---|---|---|
| ビジネス・業務理解 | 背景を掴む | 自社・競合・顧客・外部環境・現場業務 |
| KGI/KPI/CSF整理 | 成功条件を定義 | 目標、成功要因、管理指標をつなぐ |
| KPIツリー整理 | 指標構造を分解 | KGIから下位KPIへ因果で分解 |
| As-Is / To-Be | 現状と理想の差を把握 | 何が足りないかを見える化 |
| 課題整理 | 打ち手候補を決める | 課題定義、構造化、優先順位付け |

### 4. 要件定義フレーム

要件定義は、**改善に向けてどう使ってもらうかを決める工程**です。

| ステップ | 役割 | 具体内容 |
|---|---|---|
| 想定ユースケース策定 | 利用者視点を固定 | 誰が、いつ、何のために見るか |
| 具体アクション想定 | 行動に接続 | 見えた結果から何をするか |
| 全体構成整理 | 画面の役割を分ける | 全体サマリー、テーマ別、詳細分析 |
| 範囲と順番決定 | スコープ管理 | 何から作るか、何は後回しか |
| 構成要素整理 | 必要情報を定義 | KGI/KPI項目、属性項目、行動項目 |

### 5. 分析設計フレーム

本書の核はここです。分析設計とは、**「何を分析できれば意思決定に役立つか」を設計すること**です。

| 観点 | 問い | 設計対象 |
|---|---|---|
| ビジネス成果から逆算 | 何が分かれば次の判断ができるか | 問いの設定 |
| 課題から逆算 | どの課題を特定したいか | 指標の候補 |
| 比較で意味を与える | 何と比べると解釈できるか | 比較軸、比較方法 |
| 使い方に合わせる | 誰がどの粒度まで扱えるか | 情報量、難易度 |

### 6. 四つの分析タイプ

| 分析タイプ | 何を明らかにするか | 主な用途 |
|---|---|---|
| 現状診断型 | 今どうなっているか | KPI監視、異常検知 |
| 課題特定型 | どこに問題があるか | 部門別・商品別・地域別比較 |
| 特徴・特性探索型 | なぜそうなるのか | 原因候補、行動特性探索 |
| 戦略・施策評価型 | 打ち手は効いたか | 施策評価、改善判断 |

### 7. デザイン4ステップ

| ステップ | 目的 | キー原則 |

| インタラクティブ機能デザイン | 対話性を上げる | フィルター、切替、ツールヒント |

### 8. データ準備フレーム

| ステップ | 判断軸 | 重要ポイント |
|---|---|---|
| 指標確認 | 何をどう計算するか | 計算ロジックを明文化 |
| 粒度検討 | どこまで細かく持つか | 柔軟性と性能のトレードオフ |
| テーブル設計 | 何をどう持つか | チャートから逆算 |
| テーブル作成 | どう加工するか | 期間絞り、集約、不要列削減 |
| 更新ルール化 | どう運用するか | 更新頻度、更新責任、手動運用の明文化 |

### 9. 運用定着フレーム

| 領域 | 目的 | 具体策 |
|---|---|---|

## 章別エッセンス

### 第1章: ダッシュボードの種類と課題
**核心命題**: ダッシュボードは「複数のチャートを並べた画面」ではなく、ビジネス課題に対して意思決定を促す装置である。  
**反直感的洞察**: 「見える化」だけでは価値がなく、目的・意思決定・継続利用の3条件を満たさないと使われない。  
**事例サマリー**: KPIモニタリング、戦略・方針立案、施策効果測定の3タイプに整理し、それぞれに必要な深さが異なることを示す。  
**行動指針**:

- 目的が複数あるダッシュボードは分割を疑う。
- 「この画面を見たあと何を決めるか」を先に言語化する。
- 使われない原因を、設計・データ・運用の3層で切り分ける。

### 第2章: ダッシュボード構築プロジェクトの全体像
**核心命題**: ダッシュボード構築は、可視化作業ではなくプロジェクトである。  
**反直感的洞察**: BIツールの知識よりも、体制・進め方・成果物管理のほうが失敗率に効く。  
**事例サマリー**: 要求定義から運用までの5フェーズ、必要スキル、役割、会議体、ウォーターフォール/アジャイル/ハイブリッドの選び方を整理。  
**行動指針**:

- 体制図、ロードマップ、タスク表、課題表を最初に用意する。
- プロジェクト特性に応じて進め方を選び、混ぜるなら意図的に混ぜる。
- 会議体は後回しにせず先に設計する。

### 第3章: 要求定義・要件定義
**核心命題**: 良いダッシュボードは、良い要求定義と要件定義からしか生まれない。  
**反直感的洞察**: 何を表示するかより、誰がどの業務でどう使うかのほうが先である。  
**事例サマリー**: KGI/KPI/CSF、KPIツリー、As-Is/To-Be、課題整理、想定ユースケース、全体構成、構築優先順位を定義。  
**行動指針**:

- KGIだけでなくCSFまで掘り下げる。
- 想定アクションが曖昧な要件は採用しない。
- 全体サマリー、テーマ別、詳細分析を分けて考える。
- ダッシュボードが不向きならアドホック分析を選ぶ。

### 第4章: ダッシュボード設計
**核心命題**: 分析設計とは、問い・指標・比較軸を通じて意思決定の質を設計することである。  
**反直感的洞察**: 「見た目が整っている」だけではダメで、問いが浅いと美しいが役に立たない画面になる。  
**事例サマリー**: 分析タイプ、比較方法、ダッシュボード詳細設計書、データ調査の3層、問いの仮説立てまで体系化。  
**行動指針**:

- ダッシュボードの目的 -> 課題 -> 問い -> 分析要件の順で分解する。
- 指標は結果だけでなく過程も見る。
- 比較は「大きさ」「変化」「構成比」を基本に設計する。
- データ調査を設計の前倒しで行う。

### 第5章: ダッシュボードデザイン
**核心命題**: デザインは装飾ではなく、ユーザーの認知負荷を下げて分析の流れを作るためにある。  
**反直感的洞察**: 派手な可視化より、シンプルなチャートと明確な配置ルールのほうが使われる。  
**事例サマリー**: 1画面/縦長、Z型/F型視線誘導、チャートエリア、配色ルール、シグナル/ノイズ、インタラクティブ設計を整理。  
**行動指針**:

- まず棒グラフ・折れ線・数値・表の基本を使い切る。
- 分析の順番と視線の流れを一致させる。
- 複雑なチャートは分解する。
- フィルターの適用範囲を配置で伝える。

### 第6章: データ準備・ダッシュボード構築
**核心命題**: ダッシュボードの実装品質は、データの粒度・テーブル設計・更新ルールで決まる。  
**反直感的洞察**: BI上で何でも頑張るより、チャートから逆算したマート設計のほうが重要。  
**事例サマリー**: 指標ロジック、粒度、ディメンション/メジャー、マートテーブル、EAV/Tidy/Wide Spread、更新ルール、性能対策を提示。  
**行動指針**:

- チャート要件から必要カラムを逆算する。
- 粒度は細かいほど柔軟だが重い。必要十分を選ぶ。
- 同じ変換を何度も使うならDB側に寄せる。
- 数値整合性と性能確認を必須工程にする。

### 第7章: 運用・レビュー・サポート
**核心命題**: ダッシュボードはリリースしてからが本番であり、使われ続ける仕組みを設計しないと死ぬ。  
**反直感的洞察**: サポート資料やQ&Aの場も、ダッシュボード本体と同じくらい重要なプロダクトの一部である。  
**事例サマリー**: 機能レビュー、デザインレビュー、数値整合性レビュー、テスト運用、導入後効果レビュー、説明会、改善・保守を定義。  
**行動指針**:

- 構築前後でレビューの種類を分ける。
- 利用頻度・閲覧者・不便点を定点観測する。
- 使い方をユーザーに委ねず、説明会と資料をセットで渡す。
- マスタ更新と性能劣化を定期メンテ対象にする。

## 事例から抽出した原則

### 原則1: ダッシュボードは「問いの器」である

良いダッシュボードは、単に指標を並べない。  
「今どこが悪いのか」  
「その原因はどこにありそうか」  
「次に何を試すべきか」  
という問いの流れを作る。

### 原則2: 一枚に詰め込むほど価値が上がるわけではない

全体サマリー、テーマ別、詳細分析の役割分離が重要。  
一画面に詰め込みすぎると、分析の自由度ではなく認知負荷が増える。

### 原則3: 先に定義すべきはチャートではなく意思決定

チャート形式は後工程。  
先に決めるべきは、`誰が、どの業務で、何を判断するか`。

### 原則4: 設計はユーザー理解、実装はデータ粒度で決まる

設計失敗の典型はユーザー理解不足。  
実装失敗の典型は粒度ミスとマート設計ミス。  
この二つを外すと、見た目が綺麗でも使われない。

### 原則5: 運用しないダッシュボードは死蔵資産になる

更新停止、性能悪化、指標定義の不明瞭さ、サポート不足。  
これらは全て「使われないダッシュボード」を生む。

## 批判的視座

本書の強みは、ダッシュボード構築を**業務設計・分析設計・データ設計・運用設計を含む総合実務**として整理したことにあります。一方で、いくつかの前提もあります。

第一に、本書はかなり実務的であるぶん、方法論がやや重い。小規模チームやスピード最優先の現場では、ここまで丁寧な成果物整備を毎回やるのは負荷が高いでしょう。  
第二に、著者らの文脈がTreasure Dataを用いたエンタープライズ案件に根ざしているため、データ基盤やプロジェクト運営の前提が比較的大きい。スタートアップや少人数チームでは、同じ型をそのまま適用すると過剰設計になる可能性があります。  
第三に、本書は「使われる」ことを重視する一方で、組織政治や権限闘争、データオーナーシップの摩擦のような現実には深く踏み込んでいません。実務ではこの非技術要因が最難所になることも多いです。

それでも本書は、**ダッシュボードを画面設計ではなく意思決定設計として扱う**という視点を与えてくれる点で価値が高いです。

## 原著への架け橋

この圧縮読本では、本書の価値を最もよく表す**フレームワーク、判断順序、チェック項目**を優先して残しました。その代わり、原著に多く含まれる図版、具体例、レビューの運び方、チャート選択のニュアンス、データ粒度やテーブル設計の細かなケース分けはかなり圧縮しています。

次の読者は原著に当たる価値が高いです。

- 実際にダッシュボード構築案件を持っている人

逆に、ダッシュボード設計の全体像と実務の勘所を短時間で掴みたいだけなら、この圧縮版でかなり足ります。原著は「読む本」であると同時に「現場で横に置く本」でもあります。
