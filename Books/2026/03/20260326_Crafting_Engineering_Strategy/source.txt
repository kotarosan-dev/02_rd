As someone who writes books, I am continually grateful that there are people who buy a book and then actually read that book. It's equally clear to me that there's a second, large group who like the idea of books, but would much prefer a project-based approach to learning the book's content.

As the technology sector applies large language models (LLMs) to everything, one very valid reaction for authors is to be concerned. Those concerns are merited: it is possible that fewer people will

be reading books a decade from now than are reading them today. However, I also think that LLMs are a powerful tool for supporting project-based leaders.

This companion to Crafting Engineering Strategy aims to do exactly that, showcasing a number of ways you can actively engage with this book as your guide and copilot in creating engineering strategy.

There are many things that LLMs are not particularly good at, and generating a great engineering strategy is one of them. If you ask an LLM to generate your company's engineering strategy, I'm quite confident you are going to be disappointed with what it generates. However, I I am extremely confident that an LLM can be an excellent companion that can help you develop an effective engineering strategy. Used thoughtfully, an LLM can generate and analyze systems models, help with writing, and point out where you need to reinforce your strategy. This companion walks you through techniques to accomplish each of those things, starting with configuring an environment to collaborate with an LLM of your choice as you work through Crafting Engineering Strategy.

Prompting LLMs Effectively

If you want thorough coverage of working with LLMs, I recommend Chip Huyen's AI Engineering: Building Applications with Foundation Models (O'Reilly, 2024). However, most folks get started with LLMS by building their intuition chatting with them directly, which I've found rather effective. On the assumption that you've spent some time using LLMs already, I will limit myself to emphasizing three particularly valuable techniques: thinking of your LLM as a newly hired intern, improving prompts via metaprompting, and using in-context learning (such as Crafting Engineering Strategy example strategies) to improve output.

The closest reference point most folks have for prompting is using a search engine like Google. Search indexes are powerful creatures, and generally you can get exactly what you want by typing in a few uncommon words related to your topic. Seasoned searchers might often know the resource they want to retrieve the content from: for example, reddit best restaurants sf.

However, LLMs generally don't perform well with these sorts of terse prompts. For example, let's look at two prompts for creating a Python command-line tool that annotates a Markdown image by adding a description underneath it.

The first prompt uses the "LLM as search" mental model, providing a very general prompt under the impression that you are retrieving the response from an existing data source:

Prompt: I need a Python function to add image descriptions to Markdown. Description should be in a paragraph tag beneath the image.

Using ChatGPT-40, this generates a Python script, but it doesn't really generate the right Python script. This is evident solely from the function definition:

def add_image_descriptions(markdown text, description_func=None):

I wanted a script that used the existing image definition within the Markdown image specification, but because I made a general query -as if I were retrieving an existing answer-I got the wrong thing entirely.

Now let's try a second, much more verbose prompt, taking an approach I would describe as the "LLM as intern" mental model. Here, you are providing very detailed instructions, as if you were writing a ticket for a newly hired intern starting their first job writing software:

Prompt: Write a Python3 function with the definition: def add_image_descriptions(text: str) -> str:

This function should use Python's re library to extract the description text for Markdown images, and add it as floating text beneath the image, for example it should replace this:

It should do that for every Markdown image in the text. It should not do that for any Markdown links that start with [, only images which start with ![.

The second prompt took much longer to write, but it consistently generates exactly the output that I am looking for. Metaprompting

While you should work at improving your prompts yourself, LLMs can also help here, through a technique called metaprompting. The idea here is to ask the LLM to improve your prompt rather than directly following the prompt's directions.

For example, OpenAl's guide on metaprompting suggests this metaprompt to improve prompts that attempt to classify news, emphasizing that the improved prompt should create tags describing the content, provide sentiment analysis, and categorize the type of news included:

Prompt: Improve the following prompt to generate a more detailed summary. Adhere to prompt engineering best practices. Make sure the structure is clear and intuitive and contains the type of news, tags and sentiment analysis. (simple_prompt}

Only return the prompt.

Replace (simple_prompt) with your own query; no need to enclose it in quotes or otherwise, and the metaprompt will work its magic. The most effective metaprompt will vary, depending on the sort of

In-Context Learning

The final technique I'll discuss in this section is in-context learning (ICL), which is essentially providing examples of what you want the LLM to generate for you. Depending on the number of examples provided for ICL, it might be described as one-shot ICL (one example), few-shot ICL (roughly 10 or fewer examples), or many-shot ICL (many dozens to low hundreds of examples). These examples help prime the LLM to understand exactly the sort of response you want. The more examples you include, the better the prompt will typically perform. I've found that ICL solves most problems that metaprompting alone cannot solve.

You can use this book to support generating ICL examples, then include those examples in your prompts to better generate your desired output. With the book in your context window, your prompt for generating examples can be straightforward. For example:

Prompt: Provide 10 examples of diagnoses from Crafting Engineering Strategy's example strategies. Format them as a bulleted list of examples.

Using Tools via MCP

In the rare scenarios where metaprompting and ICL are insufficient to make a prompt work effectively, you can try providing purpose- built tools to your LLM that allow it to do what it does best- manipulate textual representations-while avoiding the sorts of tasks it's worse at, such as math or rendering images with significant textual content.

Chapters 4 and 5 of this companion dig into generating systems models and generating Wardley maps with LLMs. Both include approaches that rely on using Model Context Protocol servers to access additional tools. Although all major LLM providers intend to support MCP servers in the future, as things stand in mid-2025

QAa **⠀

as I write this, Claude Desktop is the only tool that provides straightforward support for allowing your project to interact with custom MCP servers. MCP's instructions for setting up your local Claude Desktop to use custom tools are the best available.

If you are wholly uninterested in running Claude Desktop but do want to experiment with MCP servers, consider the OpenAI Agents SDK. Using it will require writing custom code-but the exact sort of code that an LLM is effective at writing, if you include the relevant documentation in your prompt.

Chapter 2. Cowriting Strategy

With your environment working, the obvious task to start with is cowriting a strategy document with an LLM. A good strategy document is not just readable, but also has a clear view of your current challenge and how to address it.

Are LLM-Optimized Books a Gimmick?

Many authors are concerned about LLMs stealing their work and their livelihood. This is not an abstract concern, but a very real one: web traffic appears to be shifting toward LLMS and away from traditional web search, as folks get answers directly from LLMs that were trained on those sites' content without compensating the authors.

As web search became pervasive in the 1990s, there were similar debates about fair use. Eventually the courts decided on the exact parameters of fair use laws in the context of web crawlers, and the debate faded as websites learned to take advantage of the reach created by search engines. The status quo was not preserved-the newspaper industry in particular was irrevocably changed-but a new status quo was eventually established. Today, LLMS are in a

QAa **:

similar moment, where authors need to discover the opportunity that LLMs represent while also grappling with the reality that they may significantly change how books are read and sold. Working with O'Reilly to release the LLM-optimized edition of Crafting Engineering Strategy, along with this AI Companion, is a joint experiment in figuring out what might work. The market for LLM- optimized books is essentially nonexistent today; however, figuring out how paid written content will coexist with LLMs is a problem that some of the most thoughtful people I know have mentioned struggling with, and that I think this approach solves well.

The most extreme version of this future is one where your entire personal library is accessible to LLM agents running on your behalf. Another vision is that reading behavior doesn't change much, with humans mostly reading on our own behalf. Although I'm not sure which of those futures we'll be living in next year or next decade, my best guess is a bit of both.

While this approach has its imperfections, I'm personally finding

it useful today, and I hope it makes it easier to write effective engineering strategies for your organization.
