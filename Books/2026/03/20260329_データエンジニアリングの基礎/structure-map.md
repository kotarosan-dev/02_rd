# 構造マップ：『データエンジニアリングの基礎』

## 全体像

本書は、個別ツールの教科書ではなく、**データエンジニアリングライフサイクル**を軸にした全体設計の本である。  
構成は大きく3部と付録に分かれる。

- **第I部**: 概念と判断軸を作る
- **第II部**: 生成から提供までの各ステージを掘る
- **第III部**: セキュリティと未来予測で全体を閉じる
- **付録**: シリアライズ、圧縮、クラウドネットワークなど基盤知識を補う

## 章の流れ

### 第I部: まず「考え方」を作る

第I部は、本書の心臓部である。  
ここでライフサイクル、底流、アーキテクチャ原則、技術選定軸をまとめて頭に入れておくと、第II部以降を単発知識ではなく「どのステージの話か」「どの底流が効いている話か」で整理できる。

### 第II部: 実務の流れに沿って掘る

第II部は、現場の仕事順に近い。

1. ソースでどう生まれるか
2. どこにどう保存するか
3. どう持ってくるか
4. どう価値に変えるか
5. 誰にどう届けるか

この順番で読めば、局所最適ではなく全体最適で考えやすい。

### 第III部: 横串を締め直す

第III部は、最後にセキュリティと未来を置く。  
これは付け足しではなく、ライフサイクルを現実の組織に接続する章である。

- 10章は「守れない基盤は価値を届けられない」と確認する章
- 11章は「今の設計がどの方向へ進化しうるか」を示す章

## 重要論点の分布

| 論点 | 主に読む章 | 補足 |
|---|---|---|

| データ品質の源流 | 5章、9章 | 上流設計と下流信頼はつながっている |
| ストレージ選択 | 6章 | 性能とコストの土台 |
| バッチ/ストリーム判断 | 7章、8章、11章 | 現在の設計と未来像を往復すると理解しやすい |
| データモデリング | 8章 | Inmon、Kimball、Data Vault、ワイドテーブル |
| セルフサービスと信頼 | 9章 | 提供の失敗を減らす視点 |
| セキュリティ実務 | 10章 | 習慣・監視・最小権限が中心 |
| ライブデータスタック | 11章 | 予測として読む |

## 読み返しガイド

- 設計レビュー前に読むなら: `2章` `3章` `4章`
- 取り込み基盤を見直すなら: `7章`
- モデルやSQLが混乱しているなら: `8章`
- ダッシュボードが使われないなら: `9章`
- 認証情報や権限管理が不安なら: `10章`
- 将来の方向性を議論したいなら: `11章`

## 一番重要な骨格

本書の骨格を3行で言うなら次の通り。

1. データエンジニアリングはライフサイクル全体で考える
2. 各ステージには底流が通っている
3. 技術選定は常にユースケース、制約、価値との対応で決まる
