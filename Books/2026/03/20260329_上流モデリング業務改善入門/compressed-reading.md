## 読書ナビゲーション

本書は「付箋と模造紙で業務を可視化し、プロセスモジュール単位でつなぎ、UMLアクティビティ図と業務定義書に清書する」という**実務手順**と、「業務フローを問題発見・問題解決とコミュニケーションの共通言語にする」という**組織論**の二層構造です。第1部でモデリングの型を体で覚え、第2部で会議運営・問題の捉え方・風土（言える化）まで広げます。2010年前後のIT・UML文脈（JIS化言及等）があり、ツール名や標準の流行は時代要補正ですが、**入力／出力の受け渡し**と**部門横断の対話**という核は現場でそのまま効きます。

**読み順の目安**：いきなりツールを使うなら第1章→第2〜4章。会議が壊れている組織は第5章から入り、問題の言語化を整えるなら第6章→第7章、風土と持続性が課題なら第8章→第9章。おわりには「現場の判断実行」という経営トレンドとの接続なので、導入プロジェクトの提案書の**上乗せメッセージ**として使える。

## コアフレームワーク

### 全体の流れ（第1部の骨格）

```text
経営・組織戦略の確認
    → キックオフ（目的・ゴール共有）
    → 業務棚卸（区分表 → 一覧表）
    → 業務プロセス抽出（付箋・1プロセス1付箋）
    → ラフモデル（模造紙：タテ＝時間、ヨコ＝部門）
    → 業務プロセス記述書でビフォーモデル
    → チェック・AND/OR見直し → アフターモデル
    → UMLアクティビティ図 ＋ 業務定義書
```

### プロセスモジュール（業務プロセス記述の最小単位）

1. 業務プロセス名称  
2. 入力（インプット）と出力（アウトプット）  
3. 制約条件  
4. 開始条件  
5. 開始タイミング（リアルタイム／バッチ）  
6. 処理内容（プロセス）  
7. 帳票・情報システム  
8. 処理時間  

接続時に加える情報：

9. 伝達手段  
10. 業務区分（役割）  
11. 並列処理  
12. 例外処理・差し戻し  
13. 粒度の揃え方  
14. ルール（出所含む）  
15. 「タテ」情報（決裁・承認）  
16. **1記述書＝1業務プロセス（動詞は1つ）**

**16項目と典型ミス（復習用）**

| # | 観点 | 起きがちなミス |
|---|------|----------------|
| 1 | 名称 | 部門ごとに別名のまま |
| 2 | 入出力 | 「社内」止まりで保管場所不明 |
| 3 | 制約 | 締切は口頭のみ |
| 4 | 開始条件 | トリガーが「来たら」一語 |
| 5 | タイミング | バッチ周期が曖昧 |
| 6 | 処理 | 説明文が長く動詞不明 |
| 7 | 帳票／IT | システム名が泛称 |
| 8 | 時間 | 熟練者ベースのまま未更新 |
| 9 | 伝達手段 | システム経由で「なし」扱い |
| 10 | 役割 | 職名だけで担当範囲不明 |
| 11 | 並列 | 片系統しか描かない |
| 12 | 例外 | 差し戻し先が「上司」だけ |
| 13 | 粒度 | 新人付箋とベテラン付箋が混在 |
| 14 | ルール | 暗黙の了解のまま |
| 15 | タテ | 決裁が図から消える |
| 16 | 分割 | 「して、して」が1枚 |

### AND／OR理論（第3章）

- **AND**：複数プロセスの成果が揃うまで次工程に進めない。誰が「揃える責任」を持つか（前工程／後工程）で役割とミス耐性が変わる。  
- **OR**：どちらか一方で次へ進める方式。スピード優先になりがちで後工程が複雑化しやすい、と注意喚起。

### 問題整理の8カテゴリ（第5〜7章で横断使用）

| # | カテゴリ | 指すもの（要約） |
|---|----------|------------------|
| 1 | 戦略 | 理念・計画・事業の位置づけ |
| 2 | 収益 | 売上・コスト・利益 |
| 3 | プロセス | 前後工程を含む流れ |
| 4 | 人と組織 | 役割・風土・意思決定・ES等 |
| 5 | 仕組み | 規程・制度・システム |
| 6 | CS | 顧客視点 |
| 7 | ブランド | イメージ・無形資産 |
| 8 | 品質 | 経営品質・業務・製品サービス品質 |

### 問題発見の5ポイント（第6章）

**目的・視点・視野・時間・場所**——相互に影響し、手段が目的化したり、部門で見え方が割れたりする典型を事例で示す。

### 現象／問題／原因／課題（第6章）

- **現象**を**問題**と取り違えると打ち手がずれる（例：売上低下に対し訪問強化だけ）。  
- **課題**＝本質原因まで踏み込み「解決しなければならない」と認識が伴った論点、と定義。  
- 5回のWhy、コンサル流の「So what?／Why so?」に触れる。

### 企業活動の「ハード／ソフト」（第7〜8章）

- **ハード**：規程・システム・分業など目に見える仕組み。  
- **ソフト**：コミュニケーション・信頼・暗黙知の扱い。  
- **業務プロセス**は経営と現場の**架け橋**。フロー整備だけでは風土は変わらないが、共同作成はソフトを動かすレバーになりうる。

### 「発見」と「顕在化」（第8章）

問題を見つけても共有・意思決定に載らなければ組織の問題としては顕在化していない。業務フローは**発見**を助け、**顕在化**はコミュニケーションと心理的安全性に依存する、と線引きする。

## 章別エッセンス

### 本書の構成と読み方

**核心命題**：第1部は「書く」、第2部は「使う」。  
**行動指針**：自目的（標準化・システム・クレーム対応等）に合わせて起点章を選ぶ。

### 第1章　業務フローの書き方

**核心命題**：部門会議の「なすり合い」を、フロー上の受け渡しとデータ不整合（品番・DB・住所等）に落とすと原因が特定しやすい。  
**反直感的洞察**：ミスの直接担当部署が悪いとは限らない；インターフェースとマスタ設計が本体。  
**事例サマリー**：通信販売A社の納品トラブル → 目的（売上と品種維持）／ゴール（納品トラブルゼロ＋期日）の言語化 → 棚卸範囲（注文〜納品） → 区分表・一覧表 → 付箋ラフ → 業務プロセス記述書でビフォー → チェック → UMLサンプルまで一連の見本。  
**表1問題概要**：戦略・在庫・納期・システム・責任分界・CS・ブランドが同時に絡むことを示し、第6章8カテゴリへ接続。  
**行動指針**：問題整理表を作ったら必ず**フローに落とす**（社長セリフの通り）。コンサルはファシリテーションで整理と時間軸の確認を行う。

### 第2章　業務モデリングの準備

**核心命題**：モデリングは経営戦略・組織規程と接続しないと方向がズレる。業務プロセスは経営の手段であり、組織変更・人事評価・境界部門の影響まで含めて先に地図を読む。  
**核心手順**：キックオフ → アイスブレイク → 個別ヒアリング → 整理・ゴールイメージ → マニュアル・規程精査（ファシリテータの裏作業）→ メンバー内共有 → 全体ディスカッション（個別の「ここだけの話」は本人発言時に後押し）→ 目的・ゴールの文言化・社内広報。  
**棚卸の7目的**：効率化・標準・システム分析・BPO切出し・組織変更・プロセス変更・ミス予防。効率目的では工数・ボリューム・KPI・マニュアル整合がセットで出る。  
**障害**：目的未共有・時間・出力の統一性欠如（区分・名称・粒度・理想と現状の混在・棚卸からの脱線）。「棚卸も仕事の一つ」と重要度で位置づける。  
**業務一覧の作り方**：部門ごとに大項目を洗い出し境界で切る → 区分表を他部門と突合 → 大中小で階層化し小項目まで具体化。他人部門を自分の業務に書き忘れ・取り違えが起きやすい。  
**行動指針**：参画メンバーは実務者＋ミドル＋非正社員も可；UML詳細は必須としない。トンガリ人材・問題意識の高い人も歓迎と明記。

### 第3章　業務モデリング

**核心命題**：ラフは「つながり習得」、ビフォーは**モジュール情報の充足**、アフターは**全体整合とAND/OR**で完成。3段階を踏むのは遠回りに見えるが、粒度とプロセスモジュール理解のための訓練である。  
**反直感的洞察**：業務一覧は部門別だと隣が見えない；フロー化で「他部門の処理」が正しく分離される（営業の「確認」が法務の仕事である等）。  
**チェック**：モジュール16項目、部門跨ぎ、複雑分岐、並列、**自分ループ**（自分の出力が自分に戻る疑似ループ）、伝達手段の漏れ。例外は「比較対象（神様）」と一致判定が必要、と図解趣旨。  
**AND事例（要約）**：営業が揃う前に営業事務へ渡すと、事務側が不足書類の記憶管理に溺れる。ノルマ基準を「必要書類が揃った後」に変え、未揃いは受け付けないルールへ——**前工程が揃える**典型。  
**行動指針**：多動詞は分割（例：承認と保管を分ける）。熟練者は粗く、新人は枝葉に寄りがちなので、粒度ルールを事前共有。

### 第4章　UMLによる業務フローの記述

**核心命題**：UMLアクティビティ図はスイムレーンで役割を表し、並列も表現できる；詳細は**業務定義書**に逃がす。アクティビティはアクションの束ね、ノートでルール・タイミング・帳票を補足する。  
**比較**：フローチャート（並列弱・書き手差）、産能式（記号多・海外非互換）、BPMN（浸透・コスト）、IDEF（古典）と対比。UMLは国際標準・オブジェクト指向実装との対応・並列表現・表記統一を強調（2010年時点のJIS化言及あり）。  
**記号の最小セット**：開始／終了ノード、アクション、遷移、分岐マージ（ひし形＋ガード）、フォーク／ジョイン（並列）、スイムレーン、ノート。  
**業務定義書**：誰から／インプットアウトプットの帳票・システム／誰へ／ルール／タイミング／処理時間等、フローと1対1の番号対応。内部統制文脈では業務記述書と呼ばれる場合も。  
**行動指針**：中項目で**コンポーネント化**し、業務間の共通プロセスを見つけて標準化・改善に転用。アフターモデル清書時に粒度を再調整。

### 第5章　業務フローを活用するための準備

**核心命題**：会議が決まらないのは、場のコントロール・目的・ゴールの欠如と、事実でない発言の連鎖が典型。対策会議の対話劇は、コスト圧・海外生産・検査・開発の責任転嫁が同時多発し、議長が「来週整理」で終える失敗例。  
**ファシリテータ**：事実／感想／推測の区別（「それは事実ですか？」）、当事者化、発言タイプへの対応、利害調整、8カテゴリで発言を束ねる。議長＝ファシリテータが時間内にゴールへ収束させる。  
**発言タイプ（早見）**：断定決め付け型→根拠質問で中断；自己解決型→理由と仮説の浅さを晒す；サジ投げ／自分ワールド→具体と当事者化；言いっぱなし→「あなたならどうする」；同時多発→要約して再送；大音量・流され型にも配慮。  
**コラム**：フォーマルな場（結論必須）とインフォーマルな場（共有・相談）の分離。コミュニティ・オブ・プラクティスに近い「場」の語り。

### 第6章　問題発見・整理と問題解決

**核心命題**：複雑系の問題は5視点と8カテゴリで「誰の・どの範囲・いつの」話かを揃える。広辞苑的「問題」定義に触れたうえで、**あるべき姿固定**より変化への追従を重視。  
**現象と問題**：在庫切れは現象であり、立場によって「問題」化の仕方が違う。現象に対策すると打ち手がずれる。  
**事例**：出版社B社（売上→訪問／研修の誤対応）、製造業C社とCIO視点の迷走（目的すり替え・部門最適・コード運用の負債化・将来リスク）。社長の多目的要求で優先順位が崩れる描写。  
**8カテゴリ詳義（超要約）**：戦略＝理念・計画・現場計画；収益＝売上／コストセンター；プロセス＝前後工程つなぎ；人と組織＝ES・風土・意思決定；仕組み＝規程・IT；CS；ブランド（無形含む）；品質＝経営・業務・ISO等。  
**行動指針**：内部統制の3点セットが目的化しないよう、常に経営・現場の**原点目的**に戻す。手段目的化・目的すり替え・目的見失いの三パターンをチェック。

### 第7章　業務フローを用いた問題への取り組み

**核心命題**：ロジックツリー等の思考訓練がなくても、フローは5ポイントと8カテゴリを現場言語でカバーしやすい。MECE・仮説思考への言及あり（詳細は他書委譲）。  
**モデリングの広義の5目的**：共有、問題可視化、抑止（見られる緊張＝トヨタ的）、変化対応、属人差の低減。  
**応用**：経営課題マッピング、組織・責任権限の可視化（スイムレーン＝分業範囲）、タテの稟議フローとヨコのバリューチェーン、業務改善サイクル（戦略認識→現状モデル→改善案→小さく実行→KPI・評価）、要件定義・BPO境界。  
**ピラミッド**：ビジョン／戦略→業務プロセス→仕組み・コミュニケーション→人事基盤→個人の動機、を風土で包むイメージ。  
**行動指針**：同じ業務名でもフローを細かくするとAさんBさんのやり方差が露出する→標準化の根拠にする（権限差と混同注意）。未完成品を後工程に流さないルール設計と接続。

### 第8章　問題の顕在化と組織風土・体質

**核心命題**：トヨタ流の「問題がなくても探す」は文化とセット；仕組みだけ輸入しても顕在化しない。トヨタウェイ2001に触れ、50年の蓄積を軽く真似できないとする引用趣旨。  
**発見と顕在化**：現場で見つかっても経営に伝わらない／共有されないなら顕在化していない。業務がコミュニケーションなしで回ると無責任状態になりやすい、と構造要因を述べる。  
**業務フローの副次効果**：「世界地図」を一緒に描くことで空中戦が地上戦に変わる、という顧客の声を引用趣旨で紹介。  
**氷山モデル**（スコラ・コンサルト出典を改変と明記）：ハード（制度・システム）とソフト（暗黙ルール・恐怖・指示待ち）を分け、最下層は人と人の関わり方。  
**言える化**：可視化の前に「おかしいと言える」関係。「おかしいことをおかしいと言えないことがおかしい」。  
**ハード改革とソフト改革**：やらせ／やられるの対立、BPR室と現場の分断、腹落ちしない導入の反発。図5で時間軸と落とし穴を対比。対話と「困り合える関係」がソフト側の鍵。

### 第9章　現場で書く業務フロー

**核心命題**：設計図が間違っているか、正しくても守られていないか；いずれも**設計者と実行者の分離**が典型原因。  
**反直感的洞察**：業務部・システム屋・コンサルだけが書くと、現場は「関与していない」と実態と乖離した図になる。  
**動機付け**：経営は**効果**、現場は**効率**と日々の関心事；8カテゴリを共通言語にし、「フローは手段」と品質・収益に接続する。  
**行動指針**：時間・承認・成功体験など**書ける環境**を用意；ISO等の形だけ推進は疲弊と失敗例として警告。

### おわりに

**核心命題**：不確実性の高い環境では現場の判断実行が要になる；その前提は考える習慣・環境・フロー等の支援ツールと、共有された判断基準・権限委譲。アジャイル／アジリティ語で締め、トップコミット・巻き込み・情報発信・進捗管理は本書範囲外としてコンサル問合せへ誘導。

## 判断基準表（いつ何を使うか）

| 状況 | まず使うもの | 避けがちな落とし穴 |
|------|----------------|-------------------|
| 部門間で原因が割れない | 業務フロー上で**受け渡し点**を特定し、インプット／アウトプットの不一致を列挙 | 人格攻撃・感想の連鎖（第5章） |
| ミスが増えた直後 | 現状**ビフォー**で時系列とデータ項目を固定し、変更期間と重なるか確認（第1章型） | 現象対策だけ（第6章） |
| システム要件定義 | 業務定義書まで番号対応で揃えてから画面／APIに落とす | ベンダー任せの図だけ |
| 標準化・ISO | 8カテゴリで経営と現場の**共通目的**を接続（第9章） | 「フローを書け」単体命令 |
| 並列・合流が多い工程 | フォーク／ジョインとAND条件を明示 | 片系統だけ記述して幽霊データ |
| 属人業務が疑わしい | 同一業務名で**複数フロー**が出たら統合交渉（第9章） | 権限差と真の属人差の混同 |

## 第1章ケース：納品トラブルの構造（要約）

新商品投入後に限り「違う品・住所ミス」が増えた理由を、対話から**データと期間**に紐づけて整理する流れがモデル。典型因子は、(1) 顧客DBと受注DBの非連動、(2) 品番の重複登録と在庫側の手直し、(3) 納品先の手入力、(4) 部門間の責任のなすり合い、(5) 生産／在庫／営業の情報の非対称。社長提案どおり**プロセス全体をフローに落とす**ことで、個別の言い分を統合検証できる、というメッセージ。

## パターン／アンチパターン（業務フロー運用）

| 推奨パターン | アンチパターン | 理由（本書の論旨） |
|--------------|----------------|-------------------|
| 現場が記述書を書き、部門横でレビュー | 標準部だけが図を描き現場は閲覧のみ | 実態と乖離し、守られない（第9章） |
| ゴールに日付と「何がどう変わるか」を紐づける | 「改善しましょう」だけ | 動機と優先順位が溶ける（第2章・第6章） |
| 個別ヒアリングで本音を取り、全体では本人が話すまで待つ | ファシリが内密話を暴露 | 安心感が壊れ参加が減る（第2章） |
| 未完成の受け渡しを前工程で止める（AND） | 後工程が揃いを記憶管理 | 責任と負荷が歪む（第3章事例） |
| フォーマルで決め、インフォーマルで煮る | 1つの会議で全部やろうとする | 場の性質が衝突（第5章コラム） |
| 問題をフロー上の箇所に着地させてから対策 | 部門名指しで人格議論 | 原因特定が遅れ再発（第5・7章） |

## モジュールチェック：16項目ワンライナー

記述書／レビュー時に頭のチェックリストとして使う。詳細定義は第3章本文。

1. 名称は一覧表と一致しているか。  
2. インプット／アウトプットは「どこから／どこへ」まで書かれているか。  
3. 単体プロセスの制約（締切等）があるか。  
4. 開始条件（トリガー）は列挙されているか。  
5. 開始タイミングは即時かバッチか。  
6. 処理は**1動詞**で主体が明確か。  
7. 帳票・データ・システム名は具体的か。  
8. 処理時間は把握可能か（改善の基準値）。  
9. 伝達手段（メール・システム・手渡し等）は書かれているか。  
10. 役割分担（部門／ロール）は明確か。  
11. 並列はフォーク／ジョインで追えるか。  
12. 例外・差し戻しはどこへ戻るか判るか。  
13. 粒度はチーム合意の基準に沿うか。  
14. ルールの出所（規程名等）が示されるか。  
15. 決裁・承認などタテの流れが抜けていないか。  
16. 1枚に複合動詞を詰め込んでいないか。

## 業務改善サイクル（第7章図8の言語化）

中期計画・戦略から課題を読み、組織構造と事業形態の整合を見る → **棚卸とモデリングで現状プロセスを可視化** → 現象から問題・原因を掘り、改善後フローを設計（KPIを要所に置く）→ **小さく試す**し周辺影響を見る → 無駄削除・標準化 → 評価と再モデリング。ルール違反・リスク・個人差などを比較し、モデルで全体が把握できたうえで抜本的改善に進む、という反復が骨格。

## 落とし穴マップ（初学者向け）

| 落とし穴 | 症状 | 回避策 |
|----------|------|--------|
| 付箋に動詞が複数 | つなぎ時に順序・責任が曖昧 | 動詞ごとに分割 |

| 顕在化プロセスなし | 誰も動かない | フォーマルな決裁ルートと心理的安全性（第8章） |

## 批判的視座

- **UML唯一神化のリスク**：BPMNや現行BAS等、組織標準に合わせ表記を選べる。本書の価値はUMLそのものより**モジュール思考と受け渡しの厳密化**。  
- **AND/ORの運用**：責任を前工程に寄せる処方は強力だが、現場の力関係・ノルマ設計とセットでないと再配分が空文化する。  
- **「課題」の定義**は著者独自の使い方に近い。他文献のissue／テーマとの混同に注意。  
- **事例の年代**：DB分断・電話受注中心等、デジタル化後はインターフェースの形は変わるが、**マスタ統制とインターフェース責任**の論点は残る。  
- **第2部の処方**：風土変革は長期戦であり、本書は「フローを通じた対話」という入口を示すに留まる（おわりにで自認）。  
- **産能式・IPO表現**：第4章の記述は2010年時点の文脈。現行の開示・統制要件は制度アップデート要確認。  
- **コンサル依存**：ファシリテータを社外に置く記述はあるが、社内人材育成の具体的カリキュラムは薄い。  
- **定量評価**：KPIや効果測定に触れるが、統計的なプロセス管理や実験設計までは踏み込まない。  
- **多様性**：在宅・分散・非同期協働にはほぼ言及なし；同期型会議と対面ワークショップ前提。

## 実践チェックリスト

- [ ] 目的とゴールを**一文**で掲示し、キックオフで全員が言い換えられるか  
- [ ] 業務区分表→一覧表で**名称・粒度**が部門間で一致しているか  
- [ ] 各付箋が**1動詞**か（承認と保管、等は分割したか）  
- [ ] 各プロセスの**入力元・出力先**が次工程の入力と一致しているか  
- [ ] 伝達手段・タイミング・例外戻り先が書かれているか  
- [ ] 部門境界と**ANDの責任所在**（誰が揃えるか）が明示されているか  
- [ ] 「自分ループ」や形だけの二重チェックがないか  
- [ ] UMLに載せきれない情報を**業務定義書**に逃がしたか  
- [ ] 会議で**事実**が共有されているか（感想・推測をラベル付けしたか）  
- [ ] 論点を**8カテゴリ**に貼り付け、抜けカテゴリがないか  
- [ ] 問題を**発見**した後、正式な場で**顕在化・意思決定**されるルートがあるか  
- [ ] フロー作成に**実務担当者**が初期から入っているか（設計者＝現場の原則）

## 原著への架け橋

圧縮読本から落ちるもの：**付箋・模造紙ワークの具体的手触り**、長い会話劇（第1章・第5章等）の臨場感、UML記号の細目・図版の追随読み、索引に見える用語の網羅的把握、そして**各業界向けの個別テンプレ**。演習を自分の会社データで行うには原著とワークショップ形式が最適です。また、内部統制・ISO・IPOの**制度要件レベルの記述**は概要にとどまるため、コンプライアンス用途では別資料が必要です。

## 自習ワーク（原著・付箋と併用）

1. 自部門の「大項目」を10枚以内の付箋で洗い出し、隣部門1つと突合して境界のズレを列挙する。  
2. 代表業務1本だけ選び、第3章の8項目だけを埋めたミニ記述書を1プロセスずつ作る（所要時間は推定で可）。  
3. その業務について「ORで進めている箇所」を赤付箋で示し、後工程の手戻りコストを口頭で估算する。  
4. 第5章の対策会議脚本を、自社の最近の会議に置き換えて8カテゴリに再配置する。  
5. 第6章のB社・C社のどちらかに、自社の**現象**と**本質問題**を1行ずつ対比で書く。  
6. 業務定義書の列（誰から／インプット／アウトプット／誰へ／ルール／タイミング／時間）をスプレッドシートに写し、UML図のノートとリンクさせる。  
7. 「設計図が間違い」と「実態がずれ」のどちらかを、手元の規程・マニュアルで1例ずつ探す。  
8. 経営向け1枚レポート：8カテゴリ各1行で「今四半期の論点」を書き、フロー化予定箇所に印をつける。  
9. 第8章の氷山図を白紙に描き、自職場の「水面下」を3行で書いて1つだけフォーマル会議に持ち込む。  
10. 「フローがなくても困らない」と言える職種を1つ挙げ、その職種でも**受け渡し条件**だけはフロー化する案を作る。  
11. 顧客／取引先が見えないバックオフィス業務を1つ選び、スイムレーンに外部アクターを追加したUMLラフを描く。  
12. 第1章表1と同様に、自部署の「問題概要」を箇条書き30分タイムボックスで出し、8カテゴリに振り分ける。

## 用語ミニ辞典（本文に基づく短義）

- **ラフモデル**：付箋でインプット／アウトプット中心につないだ粗いフロー。ルールやタイミングが未充足。  
- **ビフォーモデル**：業務プロセス記述書でモジュール化しつないだ現状フロー。  
- **アフターモデル**：全体チェック・AND/OR等を経て共有された完成系（UML前）。  
- **プロセスモジュール**：業務プロセスの最小記述単位（8＋8情報）。  
- **スイムレーン**：UMLアクティビティ図の役割区画。  
- **タテのフロー**：決裁・承認など上下関係の流れ。ヨコはバリューチェーン。  
- **AND／OR**：分岐条件の論理。責任の所在設計に直結。  
- **業務棚卸**：帳簿棚卸ではなく、手順・実態・役割の把握と整理（第2章定義）。  
- **8カテゴリ**：戦略／収益／プロセス／人と組織／仕組み／CS／ブランド／品質。  
- **5ポイント**：目的・視点・視野・時間・場所。  
- **課題（本書）**：根本原因まで含め解決必須と認識された論点。  
- **言える化**：問題を安全に口に出せる関係性。  
- **ハード／ソフト**：制度・システム vs コミュニケーション・風土（第7章で比喩的に使用）。  
- **世界地図**：部門の地図を持ち寄って作る全体フローの比喩。  
- **自分ループ**：差し戻しではなく自分出力が自分入力に戻る誤ったループ。
