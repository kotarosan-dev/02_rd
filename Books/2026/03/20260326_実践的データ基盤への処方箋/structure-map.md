## 第1章 データ活用のためのデータ整備

### 章の役割

データ基盤の左端と右端をつなぐ章。  
「何のためにデータを整備するのか」と「どこで品質を担保するのか」を定義する。

### 節別マップ

| 節 | 一行要約 | キーワード | 重要度 |
|---|---|---|---|
| 1-1 | 入口と出口を可視化し、データ流通の全体像を描く | CRUD表、用語集、ユースケース | ★★★ |
| 1-2 | 品質問題は下流ではなく生成元で解く | データソース、GIGO、正しいソース | ★★★ |
| 1-3 | データ生成現場を理解して業務改善につなげる | ERD、業務レイヤ、利害対立 | ★★★ |
| 1-4 | 最低限必要なソース品質はマスタ・共通ID・履歴 | マスタ、共通ID、履歴データ | ★★★ |
| 1-5 | データレイクは元データのコピーを一箇所に集める | データレイク、湖水地方パターン | ★★★ |
| 1-6 | DWHでは共通指標を分析用DBで管理する | 共通指標、SSOT | ★★★ |
| 1-7 | DWHは段階的に作り、初期から作り込みすぎない | クレンジング、スタースキーマ、早すぎる最適化 | ★★★ |
| 1-8 | データマートは用途ごとに分け、試行錯誤を速くする | ユースケース一対一、再利用、クリーニング | ★★★ |
| 1-9 | ユースケースから逆算して投資とツール整備を決める | 5W1H、意思決定マトリクス | ★★★ |
| 1-10 | メタデータが調査コストと運用コストを下げる | 説明文、アクセス履歴、生成者巻き込み | ★★★ |
| 1-11 | サービスレベルを定義し、計測して改善する | 品質水準、便利さ、安心 | ★★★ |
| 1-12 | データスチュワードが問い合わせと改善をつなぐ | バックログ、50%ルール、橋渡し役 | ★★★ |

### 第1章の核心命題

1. データ整備は「集める作業」ではなく「使われる状態を設計する作業」である。  
2. データ品質の本丸はデータソースであり、下流補正は応急処置に過ぎない。  
3. DWHやメタデータやサービスレベルは、使われる基盤を維持するための仕組みである。

## 第2章 データ基盤システムのつくり方

### 章の役割

「どの製品を選ぶか」より前に、「何をどんな特性のデータに対して行うのか」を整理する章。  
収集・蓄積・分析・制御の各層で、ツールの使い分け基準を示す。

### 節別マップ

| 節 | 一行要約 | キーワード | 重要度 |
|---|---|---|---|
| 2-1 | データ基盤の一般形と分散処理前提を理解する | コンポーネント、分散処理、分析用DB | ★★★ |
| 2-2 | 収集方式はデータソース種別ごとに違う | ファイル、API、DB、ログ、端末、フェデレーション | ★★★ |
| 2-3 | ファイル収集では完了通知・スキーマ・圧縮形式が重要 | トリガファイル、AVRO、Parquet | ★★★ |
| 2-4 | API収集は回数制限とキー期限が事故点になる | HTTPS、APIキー、有効期限 | ★★☆ |
| 2-5 | SQL収集はDB負荷を見ながら差分・並列・レプリカを使う | フェッチ、差分、インデックス、レプリカ | ★★★ |
| 2-6 | 負荷回避にはエクスポートやダンプも選択肢になる | CSVエクスポート、ダンプ、復元DB | ★★☆ |
| 2-7 | 更新ログ/CDCは高鮮度だが構成と復旧が難しい | REDO、バイナリログ、CDCツール | ★★★ |
| 2-8 | DB収集方式は予算・負荷・鮮度で選ぶ | 方式比較、使い分け | ★★★ |
| 2-9 | ログ収集はエージェントのバッファ設計が肝 | ログ収集エージェント、欠損防止 | ★★☆ |
| 2-10 | 端末イベント収集は製品利用優先、自作時は分散MQが軸 | Kafka、Pub/Sub、順序性、冪等 | ★★★ |
| 2-11 | ETL選定はコネクタ機能とデバッグ性で見る | ETL、コネクタ、ソースコード | ★★★ |
| 2-12 | データレイクは「なくさない」を最優先に設計する | オブジェクトストレージ、冗長化、クラウド | ★★★ |
| 2-13 | データウェアハウスには分析用DBを使う | オペレーショナルDB vs 分析用DB | ★★★ |
| 2-14 | 分析用DBはクラウド前提・同一クラウド優先・使い勝手重視 | BigQuery、Redshift、Snowflake | ★★★ |
| 2-15 | 列指向圧縮を理解し、分析用DBに苦手な更新をさせない | 読み飛ばし、CTAS、DROP | ★★★ |
| 2-16 | 処理が増えたらワークフローエンジンを導入する | 再実行、スキップ、可視化 | ★★★ |
| 2-17 | ワークフローは専用か相乗りかを先に決める | Airflow、運用責任、変更容易性 | ★★☆ |

### 第2章の核心命題

1. データ収集には銀の弾丸がなく、方式の選択が最初の設計品質を決める。  
2. データレイクは保全、DWHは共通化、分析用DBは抽出・集計という役割分担を崩さない。  
3. 分析用DBやワークフローエンジンは「性能」だけでなく「運用容易性」で選ぶべきである。

## 第3章 データ基盤を支える組織

### 章の役割

データ基盤を持続可能にするための組織設計・権限設計・監査設計を扱う章。  
技術だけでは解けない「使われる基盤」の条件を定義する。

### 節別マップ

| 節 | 一行要約 | キーワード | 重要度 |
|---|---|---|---|
| 3-1 | まず成熟度と土壌を客観的に診断する | アセスメント、文化、人材 | ★★★ |
| 3-2 | 組織構造はフェーズに応じて変える | 集権、分権、ハイブリッド | ★★★ |
| 3-3 | 成功には幹部支援とビジョンと変革管理が必要 | 成功要因、コミュニケーション、評価 | ★★★ |
| 3-4 | 職種と採用戦略を明確化する | CDO、スチュワード、採用計画 | ★★☆ |
| 3-5 | データ活用とセキュリティはトレードオフである | GDPR、CCPA、利活用ニーズ | ★★★ |
| 3-6 | セキュリティポリシーと基準を定義する | 機密性、可用性、完全性 | ★★★ |
| 3-7 | 権限はIAMとグループ管理で運用可能にする | Tier、グループ、監査ログ | ★★★ |
| 3-8 | 利用ルールと権限運用をドキュメント化する | 手続き、運用フロー、IaC | ★★☆ |
| 3-9 | 棚卸で不要資産と不要権限を整理する | 野良ダッシュボード、キー更新 | ★★★ |
| 3-10 | 匿名加工でリスクを下げつつ価値低下と向き合う | 匿名加工、K-匿名性 | ★★☆ |
| 3-11 | 監査は独立組織が客観的に回す | ISMS、リスクアセスメント、PDCA | ★★★ |

### 第3章の核心命題

1. データ活用は文化・組織・運用が整って初めて継続する。  
2. 権限やセキュリティは「厳しくする」だけでなく「回る形に単純化する」ことが重要である。  
3. 棚卸と監査を定期運用に組み込まない基盤は、やがて治安が崩れる。
